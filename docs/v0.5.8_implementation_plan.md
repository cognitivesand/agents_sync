## 1. How the work splits

Two concerns, two owners:

- **Stream 1 — Canonical Metadata Core** (daemon / runtime side). Produces the `metadata`
  block, content-only change detection, the DRY removal of `last_modified`/`generation` from
  `state.json`, and content-driven stamping. **Owns and publishes the `canonical.py` contract.**
- **Stream 2 — Import Rule, CLI & Rename** (import / export / CLI side). Removes the
  configurable collision strategy, renames the rule to `last_modified_wins`, wires import /
  export / collision-compare to the `canonical.py` accessors, and adapts the import ACs.

### 1.1 The one synchronization point

The import rule compares `last_modified` *read from the canonical metadata* — a value Stream 1
produces. To keep the streams independent after kickoff, **Stream 1 lands Phase 1.0 (the
`canonical.py` accessor contract) first**, commits, and pushes. That single push unblocks
Stream 2. Everything after Phase 1.0 runs in parallel.

If Stream 2 wants to start before Phase 1.0 lands, it codes against the **agreed signatures in
§1.2** and integrates once they're on the branch — but the path of least friction is: Stream 1
pushes Phase 1.0 on day one.

### 1.2 Inter-stream contract — `canonical.py` public API (Stream 1 delivers)

```python
def canonical_content(canonical: dict[str, Any]) -> dict[str, Any]:
    """The canonical document minus its 'metadata' block (deep copy). Drives the digest."""

def canonical_metadata(canonical: dict[str, Any]) -> dict[str, Any]:
    """The nested 'metadata' block, or {} when absent (old canonicals)."""

def canonical_last_modified(canonical: dict[str, Any]) -> float | None:
    """metadata['last_modified'] when present, else None. Used by export + collision compare."""

def set_canonical_metadata(
    canonical: dict[str, Any], *, last_modified: float, generation: int
) -> None:
    """Write the nested metadata block in place. Used by content-change stamping, adoption,
    and import promotion."""

def canonical_digest(canonical: dict[str, Any]) -> str:
    """UNCHANGED signature; now hashes canonical_content(canonical) ONLY — the crux that makes
    a metadata-only diff produce an unchanged digest (no reproject, no archive)."""
```

Metadata block shape (the single convention both streams use):
`canonical["metadata"] = {"last_modified": <float POSIX>, "generation": <int>}`.

## 2. STREAM 1 — Canonical Metadata Core

**Owner:** collaborator A. **Goal:** the canonical carries its own `metadata`; change detection
is content-only; `last_modified`/`generation` leave `state.json` entirely.

Per step: full `uv run pytest`, `mypy --strict`, `ruff check` green; one commit per phase;
surgical edits.

### Phase 1.0 — `canonical.py` accessors + content-only digest  ⟵ land + push FIRST

- Add `canonical_content`, `canonical_metadata`, `canonical_last_modified`,
  `set_canonical_metadata` per §1.2 (`canonical.py`, near the existing `canonicalize` @ :64 /
  `canonical_digest` @ :169).
- Change `canonical_digest` (:169) to hash `canonical_content(canonical)` only — exclude the
  `metadata` block. **This is the crux:** a metadata-only difference must not move the digest.
- Confirm `canonicalize` (:64) preserves the `metadata` block (it deep-copies unknown fields →
  it does; add an assertion/test to pin it).
- **Tests (write first):** `metadata` excluded from digest; identical content + different
  metadata → equal digest; `canonical_content`/`canonical_metadata` round-trip; `canonicalize`
  preserves `metadata`.
- **Commit, push.** Announce to Stream 2 that the contract is live.

### Phase 1.1 — DRY: remove `last_modified`/`generation` from `state.py`

- `PairState`: delete the `last_modified` and `generation` fields, `bump()` (:138), and their
  (de)serialization. Backward-compatible: old `state.json` simply has the fields ignored on load
  (do **not** error on their presence).
- Rationale check (already verified in 008): `engine.py` runtime conflict resolution does **not**
  read these fields — only export, import-collision compare, and adoption do, and each has the
  canonical on disk. Re-grep to confirm no remaining `PairState.last_modified` / `.generation` /
  `.bump(` readers outside the metadata path.
- **Tests:** loading an old `state.json` carrying the fields succeeds and ignores them; no code
  reads `PairState.last_modified`/`.generation`.

### Phase 1.2 — content-driven stamping on the canonical metadata

- The content-changed stamp that today calls `PairState.bump()` from `update_state_n_way`
  (`rendering.py` @ :321) now writes the **canonical** `metadata` via
  `set_canonical_metadata(canonical, last_modified=now, generation=prev+1)` — and **only when
  content actually changed** (a heal / reproject of unchanged content must NOT move it).
- Audit every writer for the "stamp only on real content change" invariant:
  `engine.py` (adoption / conflict / extend paths) and `canonical_projection.py`
  (`project_from_canonical` / reproject). A reproject of unchanged content leaves `last_modified`
  untouched.
- **Tests:** content change bumps `last_modified` + `generation`; heal/reproject of unchanged
  content does not; conflict resolution stamps once.

### Phase 1.3 — adoption writes metadata; daemon-side readers read the nested block

- Adoption (`engine.py` paths that call `canonical.save_canonical` @ canonical.py:213): ensure
  the canonical carries a `metadata` block before save (seed `last_modified`/`generation` if
  absent — old canonicals gain metadata on first content-driven write).
- `sync._adopt_orphan_canonicals` (`sync.py` @ :157): today reads the **top-level**
  `canonical.get("last_modified")`; change to `canonical_last_modified(canonical)` (nested
  block). Do not write the values back into `state`.
- **Tests:** adopting an orphan canonical with a `metadata` block records nothing in state for
  `last_modified`/`generation`; an orphan **without** metadata still adopts (None handled).

### Stream 1 — Definition of Done

`state.py` no longer carries `last_modified`/`generation`; stamping moves only on real content

## 5. Gotchas (both streams)

- **`tmp/` is NOT gitignored.** Never `git add -A`; stage explicit paths only.
- **Tests are excluded from the mypy gate** (`pyproject [tool.mypy] files = ["src"]`). The
  PostToolUse hook runs `mypy --strict` on edited test files and shows `no-untyped-def` noise —
  EXPECTED for tests; match the loose test style, don't "fix" it.
- **PostToolUse ruff autofix removes an unused import between edits** — add an import and its
  first use in the same edit, or re-add after.
- **Bash discipline:** one action per call (no `&&`), no `sed`, relative paths only, `~/.claude`
  via the file tools.
- **`canonicalize` preserves unknown fields** (deep copy) — adding `metadata` is safe (pin with
  a test in Phase 1.0).
- **FR-14 reproject archives tool bytes BEFORE re-render** — so a metadata-only diff must never
  enter the reproject path. The content-only digest (Phase 1.0) is what guarantees this; the old
  `xfail` wrongly asserted an archive on identical content.
- **`engine.py` runtime conflict does NOT use `last_modified`/`generation`** (verified) — that is
  what makes the DRY removal from `state.py` safe.
- **Historical docs keep the old name.** `docs/audits/*` and old `v0.4.x` plans mention
  `mtime_wins`; leave them as dated snapshots. Only live docs (architecture, README, this plan)
  use `last_modified_wins`.
