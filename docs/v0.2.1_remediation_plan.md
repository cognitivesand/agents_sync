# agents_sync v0.2.1 — Remediation Plan

This patch release corrects the safety and maintainability issues found during the code-quality audit of v0.2. The scope is intentionally narrow: prevent destructive behavior under misconfiguration or malformed inputs, add tests that lock those protections in, then do only the smallest refactor needed to keep the sync core understandable.

## Release Goals

- Fail closed when configured sync roots are missing or unreadable.
- Validate externally supplied `pair_id` values before using them for identity or filesystem paths.
- Detect slug/path collisions before any render, archive, move, or state mutation.
- Detect duplicate `pair_id`s during discovery and skip the affected items safely.
- Reduce the blast radius of `Syncer` by separating discovery and planning helpers from filesystem execution.

## Non-Goals

- No state schema migration unless a validation fix requires recording additional metadata.
- No file-level merge inside skill folders.
- No inotify / fsevents replacement for polling.
- No automatic migration from v0.1.
- No broad rewrite of parser/render behavior beyond validation and collision checks.

## Findings To Correct

| Finding | Risk | v0.2.1 outcome |
|---|---|---|
| Missing configured directories are treated as empty discovery results | A typo, unmounted home directory, or permission problem can propagate removals and archive/move the opposite side | Startup and each poll validate configured roots; missing/unreadable roots produce configuration/runtime errors and no sync mutation |
| Raw `pair_id` is used as a path segment | Malformed or hostile IDs can escape the state/archive tree | Only UUIDv4 strings are accepted as managed IDs; invalid IDs are treated as malformed artifacts and skipped |
| Slug/path collisions are not guarded | Two artifacts can overwrite or alias the same counterpart path | Collisions are detected before rendering; affected pairs are skipped and logged |
| Duplicate `pair_id`s on the same side overwrite discovery entries | One file silently wins and the other may be ignored or overwritten later | Duplicate IDs are structured errors; all entries sharing that duplicate ID are left untouched |
| `Syncer` owns discovery, planning, execution, and state mutation | Safety logic is hard to reason about and easy to regress | Extract small helpers for validation/discovery planning while preserving behavior |

## Implementation Phases

### Phase 1 — Configuration and Root Validation

Add a `validate_config(config) -> list[ConfigError]` step before `Syncer` starts. Validate that:

- `poll_interval_seconds` is positive.
- `state_path` has a writable parent or can create one.
- `claude_agents_dir`, `claude_skills_dir`, `codex_agents_dir`, and `codex_skills_dir` exist and are directories.
- Each configured root is readable; target roots are writable.

During daemon operation, discovery must distinguish "directory exists but contains no artifacts" from "directory cannot be read." A failed root read aborts the current poll before pair processing and returns a runtime sync failure signal; it must not be interpreted as user deletion.

Tests:

- Missing root on startup exits with configuration-failure code.
- Missing root during a later poll leaves files and `state.json` unchanged.
- Empty existing roots remain valid and sync as empty.
- Permission-denied root leaves files and `state.json` unchanged.

### Phase 2 — Pair ID Validation

Introduce a single `validate_pair_id(value) -> str` helper that accepts only canonical UUIDv4 text. Use it for:

- Claude frontmatter `pair_id`.
- Codex TOML `pair_id`.
- Codex and Claude skill `SKILL.md` frontmatter `pair_id`.
- `state.json` keys.
- Canonical filenames.

Invalid IDs should be treated as malformed artifacts, not foreign artifacts. The daemon logs the path and validation error, skips that artifact for the poll, and continues with other pairs.

Tests:

- `pair_id: ../escape` does not create files outside the state directory.
- Non-UUID, UUIDv1, and empty `pair_id` values are rejected.
- A malformed pair does not block valid unrelated pairs.
- Invalid state entries are skipped and logged without deleting live files.

### Phase 3 — Discovery Collision Detection

Change discovery from direct `pair_id -> PairInfo` assignment to a two-step process:

1. Collect discovered artifacts as records: `kind`, `side`, `path`, `pair_id`, `name`, `slug`, `digest`, `mtime`, and `pair_id_present`.
2. Validate records before planning sync actions.

Reject a poll item when any of these are true:

- More than one artifact on the same side has the same valid `pair_id`.
- A `pair_id` appears for both `agent` and `skill`.
- Two different pairs would render to the same target path.
- A foreign artifact would render over an existing managed counterpart belonging to a different pair.

Rejection means no archive, render, move, canonical write, or state write for the affected pair. Other independent pairs continue.

Tests:

- Duplicate Claude agent `pair_id`s are both left untouched.
- Duplicate Codex agent `pair_id`s are both left untouched.
- Agent/skill reuse of the same `pair_id` is skipped.
- Two foreign artifacts with the same slug are not adopted.
- Foreign artifact slug collision with an existing managed pair is not adopted.

### Phase 4 — Collision-Safe Rendering

Before `_render_claude` or `_render_codex` writes anything, compute the intended path and assert that it is either:

- The existing path for the same pair, or
- A non-existent path, or
- A path explicitly approved by a prior rename plan for the same pair.

If the assertion fails, log a collision and skip the operation. This check is defense-in-depth for any caller that bypasses the discovery planner.

Tests:

- Name change to an occupied slug is rejected without overwriting either pair.
- Rename collision for a skill directory is rejected without staging directories left behind.
- Failed collision check leaves canonical and `state.json` unchanged.

### Phase 5 — Focused Refactor

Keep this refactor mechanical and test-backed:

- Move root/config validation to `config.py`.
- Move artifact records and discovery validation to a new `discovery.py`.
- Keep parse/render code in `claude_io.py` and `codex_io.py`.
- Keep archive and atomic-write helpers unchanged except for validated path inputs.
- Leave `Syncer` responsible for applying already-validated sync decisions.

The target is not a new architecture. The target is that destructive operations become visibly downstream of validation.

Tests:

- Existing v0.2 end-to-end tests still pass unchanged.
- New regression tests from Phases 1-4 pass.
- A no-change poll still reports zero changes and performs no writes.

## Acceptance Criteria

- [ ] AC-1: Given any configured root is absent at startup, When `agents-sync` starts, Then it exits with the configuration-failure exit code and performs no filesystem mutation.
- [ ] AC-2: Given any configured root disappears after startup, When the next poll runs, Then the poll aborts before pair processing and no counterpart files are archived or removed.
- [ ] AC-3: Given a file contains an invalid `pair_id`, When the watcher polls, Then that file is skipped, a structured error names the file and invalid value class, and unrelated valid pairs still sync.
- [ ] AC-4: Given duplicate valid `pair_id`s are discovered on the same side, When the watcher polls, Then all artifacts sharing that duplicate ID are left untouched.
- [ ] AC-5: Given two artifacts would render to the same target path, When adoption or rename planning runs, Then the operation is rejected before archive/write/move/state mutation.
- [ ] AC-6: Given a collision or validation failure affects one pair, When other independent pairs are valid, Then those independent pairs still process normally.
- [ ] AC-7: Given user inputs are unchanged, When the daemon polls repeatedly, Then it produces no writes, no canonical updates, and no archive entries.

## Test Matrix

| Area | Unit tests | End-to-end tests |
|---|---|---|
| Config/root validation | `validate_config`, root permission checks | startup missing root, runtime missing root |
| Pair ID validation | UUIDv4 acceptance/rejection | malformed file skipped while valid pair syncs |
| Discovery collisions | duplicate IDs, kind mismatch, slug/path conflicts | foreign adoption collision, managed rename collision |
| Rendering guard | target path ownership assertion | no overwrite and no state mutation on collision |
| Regression | parser/render fixed points | all existing sync flows |

Run before release:

```bash
uv run pytest
```

