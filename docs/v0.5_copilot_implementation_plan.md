# v0.5 implementation plan - GitHub Copilot adapter

This plan implements the GitHub Copilot slice of `v0.5`, on the dedicated
branch `feat/v0.5-copilot`.

The adapter covers two GitHub Copilot surfaces:

1. **Copilot CLI user home**, rooted at `~/.copilot/`.
2. **VS Code user-profile customizations**, using VS Code's user data/profile
   customization files:
   - `*.instructions.md`
   - `*.prompt.md`
   - `*.chatmode.md` legacy files
   - `*.agent.md` custom agent files
   - user-profile `mcp.json`

The adapter intentionally avoids repository-scoped `.github/`, `.vscode/`,
`.claude/`, and `.agents/` files. Those belong to the user's projects and
should not be managed by this user-level adapter PR.

Supported v0.5 customization types:

- `agent`, via:
  - Copilot CLI / common user agents: `~/.copilot/agents/*.md` or
    `~/.copilot/agents/*.agent.md`
  - VS Code user-profile custom agents: `*.agent.md`
  - VS Code legacy custom chat modes: `*.chatmode.md` as read/adopt-only or
    migration-compatible input
- `skill`, via:
  - personal skills under `~/.copilot/skills/<name>/SKILL.md`
- `rules`, via:
  - VS Code user-profile `*.instructions.md`
- `slash_command`, via:
  - VS Code user-profile `*.prompt.md`
- `mcp_server`, via:
  - Copilot CLI `~/.copilot/mcp-config.json`
  - VS Code user-profile `mcp.json`

It does not touch GitHub account state, Copilot organization customizations,
repository `.github/` files, VS Code settings JSON except the user-profile
`mcp.json`, extension-contributed customizations, plugin packages, hooks,
workspace `.vscode/mcp.json`, or Copilot cloud agent repository settings.

## Why GitHub Copilot

GitHub Copilot is a high-value v0.5 adapter because it exposes several
file-backed customization surfaces across both the terminal and VS Code:

- Copilot CLI supports custom agents, skills, and MCP servers.
- VS Code supports custom instructions, prompt files, custom agents, Agent
  Skills, and MCP configuration.
- Copilot Agent Skills use the open `SKILL.md` directory shape and are designed
  to work across VS Code, Copilot CLI, and Copilot cloud agent.
- VS Code prompt files are Markdown slash commands with `.prompt.md`.
- VS Code custom agents are Markdown files with `.agent.md`.
- VS Code custom instructions are Markdown rules/instructions files with
  `.instructions.md`.

This adapter proves that the v0.5 registry model can support a tool with two
related but distinct local surfaces: a terminal home directory and a VS Code
profile directory. It also tests migration compatibility for legacy
`.chatmode.md` files, which VS Code now treats as the predecessor to
`.agent.md`.

## Scope

### In scope

- Register a new `copilot` agentic_tool.
- Support Copilot CLI/custom user agents from:
  `~/.copilot/agents/*.md` and `~/.copilot/agents/*.agent.md`.
- Support VS Code user-profile custom agents from:
  `*.agent.md`.
- Support VS Code legacy chat mode files from:
  `*.chatmode.md`.
- Support personal Agent Skills from:
  `~/.copilot/skills/<name>/SKILL.md`.
- Support VS Code user-profile instructions from:
  `*.instructions.md`.
- Support VS Code user-profile prompt files from:
  `*.prompt.md`.
- Support Copilot CLI MCP server slots from:
  `~/.copilot/mcp-config.json`.
- Support VS Code user-profile MCP server slots from:
  `mcp.json`.
- Per-OS defaults for the Copilot CLI half:

  | OS | Copilot home | Agents | Skills | CLI MCP |
  |---|---|---|---|---|
  | Linux | `~/.copilot/` | `~/.copilot/agents/` | `~/.copilot/skills/` | `~/.copilot/mcp-config.json` |
  | macOS | `~/.copilot/` | `~/.copilot/agents/` | `~/.copilot/skills/` | `~/.copilot/mcp-config.json` |
  | Windows | `%USERPROFILE%\.copilot\` | `%USERPROFILE%\.copilot\agents\` | `%USERPROFILE%\.copilot\skills\` | `%USERPROFILE%\.copilot\mcp-config.json` |

- Configurable paths for the VS Code user-profile half:

  | Surface | Config key | Default behavior |
  |---|---|---|
  | User instructions | `copilot_vscode_user_instructions_dir` | unset until installer/profile discovery configures it |

  | User agents | `copilot_vscode_user_agents_dir` | unset until installer/profile discovery configures it |
  | User MCP file | `copilot_vscode_user_mcp_file` | unset until installer/profile discovery configures it |

- `copilot_enabled` config key, default `true`.
- Separate optional enable flags for each half:
  - `copilot_cli_enabled`, default `true`
  - `copilot_vscode_user_profile_enabled`, default `true`
- Runtime `unavailable` status when configured roots are missing.
- Secret handling for MCP servers through the global
  `mcp_server_secret_policy`.
- Unit and integration tests covering all supported types.
- README, protocol notes, installer sample config, and requirements matrix

## Goals

1. A Copilot custom agent syncs to every other available tool that supports
   `agent`, and agents from other tools render into Copilot-compatible
   `.agent.md` files.
2. A Copilot Agent Skill syncs to every other available tool that supports
   `skill`, including auxiliary files next to `SKILL.md`.
3. A VS Code user-profile instruction file syncs to every other available tool
   that supports `rules`.
4. A VS Code prompt file syncs to every other available tool that supports
   `slash_command`.
5. Copilot CLI MCP server slots sync through `mcp_server` without touching
   sibling slots in `~/.copilot/mcp-config.json`.
6. VS Code user-profile MCP server slots sync through `mcp_server` without
   touching sibling slots in the profile `mcp.json`.
7. Copilot can be enabled on a machine without Copilot CLI or VS Code profile
   paths installed; missing roots make the corresponding surface
   `unavailable`, not fatal.
8. Unsupported and cloud/organization/repository-only surfaces are documented
   clearly so users do not expect `agents_sync` to move them.

## Non-goals

- Auto-detect every VS Code profile path on every operating system in the
  initial PR.
- Sync repository-scoped Copilot files.
- Sync organization-level Copilot customizations.
- Sync Copilot cloud agent settings from GitHub.com.
- Translate VS Code hooks or Copilot CLI hooks into any current type.
- Translate plugin packages into individual artifacts.
- Add new sync-engine behavior.
- Persist identity outside managed artifacts.

## Baseline findings

- Copilot CLI supports custom agents, skills, and MCP servers.
- Copilot CLI stores configured MCP servers in `mcp-config.json` under the
  Copilot home directory, which defaults to `~/.copilot` and can be changed by
  `COPILOT_HOME`.
- VS Code custom agents are Markdown files with the `.agent.md` extension.
- VS Code custom agents can be stored in the workspace or in the user profile.
  VS Code also recognizes `~/.copilot/agents` as a user-level custom agent
  location.
- VS Code custom chat modes were renamed to custom agents. Existing
  `.chatmode.md` files should be treated as legacy custom agent files and
  rendered as `.agent.md` for new output.
- VS Code prompt files are Markdown files with the `.prompt.md` extension and
  optional YAML frontmatter.
- VS Code prompt files are slash-command-like reusable prompts.
- VS Code instruction files are Markdown files with the `.instructions.md`
  extension, plus optional YAML frontmatter for file matching and metadata.
- VS Code user instruction and prompt files can be synced by VS Code Settings
  Sync when the user enables the Prompts and Instructions sync category.
- VS Code Agent Skills are folders with a `SKILL.md` file. Personal skills can
  live under `~/.copilot/skills/`.
- VS Code MCP server configuration is stored in an `mcp.json` JSON file. This
  file can be workspace-scoped or user-profile scoped.
- Agent Skills work across VS Code, Copilot CLI, and Copilot cloud agent, but
  this PR only manages local file-backed user-level surfaces.

## Resolved design decisions for this PR

- **D-cop-1. Two local surfaces.** The adapter has a Copilot CLI half and a VS
  Code user-profile half. They share one `copilot` agentic_tool name but use
  distinct config keys.
- **D-cop-2. Supported types.** Copilot declares exactly
  `{agent, skill, rules, slash_command, mcp_server}`.
- **D-cop-3. Repository scope excluded.** The adapter reads and writes only
  user-level locations. Repository `.github/`, `.vscode/`, `.claude/`, and
  `.agents/` paths remain owned by projects.
- **D-cop-4. Agent rendering.** New Copilot agents render as `.agent.md`.
  Legacy `.chatmode.md` files are parsed and adopted, but new output should use
  `.agent.md`.
- **D-cop-5. CLI agent compatibility.** `~/.copilot/agents/*.md` and
  `~/.copilot/agents/*.agent.md` are both parsed. New rendered output should
  prefer `.agent.md` unless compatibility testing proves Copilot CLI requires
  plain `.md`.
- **D-cop-6. Agent identity.** Copilot agent files store `pair_id` in YAML
  frontmatter. Common fields (`name`, `description`, `model`, `tools`, and
  body) map to canonical fields. Copilot-only fields are preserved under
  `per_agentic_tool_only["copilot"]` or
  `per_agentic_tool_extra["copilot"]`.
- **D-cop-7. Agent frontmatter preservation.** Preserve VS Code/Copilot fields
  such as `argument-hint`, `agents`, `user-invocable`,
  `disable-model-invocation`, `target`, `mcp-servers`, `metadata`, `handoffs`,
  and preview `hooks` without interpreting unsupported behavior.
- **D-cop-8. Skill identity.** Copilot skills use the open `SKILL.md`
  directory shape under `~/.copilot/skills/<name>/SKILL.md`. The adapter shares
  the same open-spec helper used by other skill-capable tools wherever
  possible.
- **D-cop-9. Instruction files map to `rules`.** VS Code
  `*.instructions.md` files are the Copilot user-profile counterpart to the
  protocol `rules` type. The body remains Markdown; YAML frontmatter is
  preserved.
- **D-cop-10. Prompt files map to `slash_command`.** VS Code `*.prompt.md`
  files map to `slash_command`. The prompt body maps to canonical `body`; YAML
  frontmatter maps to description/name/argument/model/tool fields where
  possible.
- **D-cop-11. Prompt command names.** Prompt command names derive from the
  frontmatter `name` when present; otherwise from the file stem. Recursive
  paths map to namespaced commands with `:` where the protocol needs
  namespacing.
- **D-cop-12. MCP identity.** Copilot MCP slots use the shared v0.5
  `mcp_server_io` convention: top-level `pair_id` inside the server slot, with
  the slot rendered through the correct shared keyed map.
- **D-cop-13. MCP file split.** Copilot CLI `mcp-config.json` and VS Code
  user-profile `mcp.json` are both supported. They must preserve unrelated
  keys and sibling server entries.
- **D-cop-14. Secret policy.** Copilot MCP secrets are handled only by
  `mcp_server_secret_policy`; the Copilot adapter does not invent a separate
  policy.
- **D-cop-15. VS Code profile paths are configurable.** The adapter should not
  hard-code fragile VS Code profile directories in core config. Installer
  helpers may populate platform-specific defaults later, but the runtime should
  treat these paths as user-configured.

## Workstreams

### 1) Copilot IO module

Target module: new `src/agents_sync/copilot_io.py`.

Implement these helpers:

- `extract_pair_id_from_copilot_agent_md(text) -> str | None`
- `parse_copilot_agent_md(text, prior_canonical, *, artifact_path, artifact_root)`
- `render_copilot_agent_md(canonical, prior_text=None) -> str`
- `extract_pair_id_from_copilot_skill_md(text) -> str | None`
- `parse_copilot_skill_md(text, prior_canonical, *, artifact_path, artifact_root)`
- `render_copilot_skill_md(canonical, prior_text=None) -> str`
- `extract_pair_id_from_copilot_instruction_md(text) -> str | None`
- `parse_copilot_instruction_md(text, prior_canonical, *, artifact_path, artifact_root)`
- `render_copilot_instruction_md(canonical, prior_text=None) -> str`
- `extract_pair_id_from_copilot_prompt_md(text) -> str | None`
- `parse_copilot_prompt_md(text, prior_canonical, *, artifact_path, artifact_root)`
- `render_copilot_prompt_md(canonical, prior_text=None) -> str`
- Copilot CLI MCP wrappers around shared `mcp_server_io`
- VS Code MCP wrappers around shared `mcp_server_io`

Agent behavior:

- Parse `.agent.md`, `.chatmode.md`, and compatible `.md` files with YAML
  frontmatter plus Markdown body.
- Derive `name` from frontmatter when present, otherwise from the filename stem.
- Strip `.agent`, `.chatmode`, or `.md` suffix artifacts from the canonical name.
- Round-trip common fields:
  - `name`
  - `description`
  - `model`
  - `tools`
  - body
- Preserve Copilot/VS Code-specific fields:
  - `argument-hint`
  - `agents`
  - `user-invocable`
  - `disable-model-invocation`
  - `infer`
  - `target`
  - `mcp-servers`
  - `metadata`
  - `handoffs`
  - `hooks`
- Store unsupported/preview fields under
  `per_agentic_tool_extra["copilot"]`.
- Render new output to `<target_slug>.agent.md`.
- Do not render new `.chatmode.md` files.
- Preserve body byte-stability where possible.

Skill behavior:

- Reuse the shared open-spec `SKILL.md` parser/renderer where possible.
- Render to `~/.copilot/skills/<target_slug>/SKILL.md`.
- Preserve auxiliary files exactly like existing skill adapters.
- Round-trip fields:
  - `name`
  - `description`
  - `argument-hint`
  - `user-invocable`
  - `disable-model-invocation`
  - `context`
  - body
- Enforce or validate Copilot skill naming constraints where safe:
  lowercase letters, numbers, and hyphens; parent directory should match the
  skill name.
- Preserve unknown frontmatter in `per_agentic_tool_extra["copilot"]`.

Rules / instructions behavior:

- Parse VS Code `*.instructions.md` files as Markdown with optional YAML
  frontmatter.
- Store `pair_id` in YAML frontmatter.
- Derive canonical `name` from frontmatter `name` when present, otherwise from
  the filename stem.
- Preserve frontmatter fields such as:
  - `description`
  - `applyTo`
  - `paths`
  - `globs`
  - `mode`
  - unknown fields
- Preserve the Markdown body verbatim.
- Render to `<target_slug>.instructions.md`.
- Do not manage repository `.github/copilot-instructions.md`.

Slash-command / prompt behavior:

- Parse VS Code `*.prompt.md` files as Markdown with optional YAML
  frontmatter.
- Store `pair_id` in YAML frontmatter.
- Map frontmatter fields:
  - `name` -> canonical name override
  - `description` -> canonical description
  - `argument-hint` -> tool-specific metadata
  - `agent` -> tool-specific metadata
  - `model` -> tool-specific metadata
  - `tools` -> canonical/tool-specific tools metadata
- Preserve unknown frontmatter in `per_agentic_tool_extra["copilot"]`.
- Preserve the Markdown body as canonical `body`.
- Render to `<target_slug>.prompt.md`.
- Avoid converting prompt files into Copilot CLI commands; the supported
  slash-command surface for this PR is VS Code prompt files.

MCP behavior:

- Parse Copilot CLI `~/.copilot/mcp-config.json`.
- Parse VS Code user-profile `mcp.json`.
- Preserve unrelated keys in both JSON files.
- Preserve sibling server entries in both JSON files.
- Normalize common MCP server forms:
  - stdio command/args/env
  - HTTP/SSE URL-based servers
  - headers and auth material
- Preserve Copilot/VS Code-specific fields such as:
  - server type
  - timeout
  - tool filters
  - trust/auto-approval metadata
  - OAuth metadata
- Let shared secret-policy code handle `env`, `headers`, tokens, and auth.

Deliverables:

- New Copilot IO module.
- Focused IO unit tests for agents, skills, instructions, prompts, CLI MCP, and
  VS Code MCP.

### 2) Agentic-tool spec registration

Target module: `src/agents_sync/agentic_tool_spec.py`.

Add `_build_copilot_spec()`:

```python
return AgenticToolSpec(
    name="copilot",
    config_dir_keys={
        "agent": [
            "copilot_cli_agents_dir",
            "copilot_vscode_user_agents_dir",
        ],
        "skill": "copilot_cli_skills_dir",
        "rules": "copilot_vscode_user_instructions_dir",
        "slash_command": "copilot_vscode_user_prompts_dir",
        "mcp_server": [
            "copilot_cli_mcp_config_file",
            "copilot_vscode_user_mcp_file",
        ],
    },
    io={
        "agent": CustomizationTypeIO(
            parse=parse_copilot_agent_md,
            render=render_copilot_agent_md,
            extract_pair_id=extract_pair_id_from_copilot_agent_md,
            storage="single_file",
            file_suffix=".agent.md",
            accepted_file_suffixes=(".agent.md", ".chatmode.md", ".md"),
        ),
        "skill": CustomizationTypeIO(
            parse=parse_copilot_skill_md,
            render=render_copilot_skill_md,
            extract_pair_id=extract_pair_id_from_copilot_skill_md,
            storage="directory_skill",
            file_suffix="",
        ),
        "rules": CustomizationTypeIO(
            parse=parse_copilot_instruction_md,
            render=render_copilot_instruction_md,
            extract_pair_id=extract_pair_id_from_copilot_instruction_md,
            file_layout=RulesFileLayout(extension=".instructions.md"),
            recursive=True,
        ),
        "slash_command": CustomizationTypeIO(
            parse=parse_copilot_prompt_md,
            render=render_copilot_prompt_md,
            extract_pair_id=extract_pair_id_from_copilot_prompt_md,
            file_layout=SlashCommandFileLayout(extension=".prompt.md"),
            slugify_name=slash_command_slug,
            recursive=True,
        ),
        "mcp_server": [
            CustomizationTypeIO(
                parse=parse_copilot_cli_mcp_server_json,
                render=render_copilot_cli_mcp_server_json,
                extract_pair_id=extract_pair_id_from_copilot_cli_mcp_server_json,
                file_layout=SharedKeyedMapLayout(
                    shared_path_config_key="copilot_cli_mcp_config_file",
                    map_key_path=("servers",),
                    file_format="json",
                ),
            ),
            CustomizationTypeIO(
                parse=parse_copilot_vscode_mcp_server_json,
                render=render_copilot_vscode_mcp_server_json,
                extract_pair_id=extract_pair_id_from_copilot_vscode_mcp_server_json,
                file_layout=SharedKeyedMapLayout(
                    shared_path_config_key="copilot_vscode_user_mcp_file",
                    map_key_path=("servers",),
                    file_format="json",
                ),
            ),
        ],
    },
    disable_config_key="copilot_enabled",
)
```

If the existing `AgenticToolSpec` cannot express multiple roots or multiple IO
entries for the same customization type, pause and fix the protocol type PR
rather than adding Copilot-specific sync-engine branching.

Then add `"copilot": _build_copilot_spec()` to `default_agentic_tools()` in
alphabetical order.

### 4) Runtime availability behavior

Target modules: no sync-engine changes expected unless multiple roots per
customization type are not yet supported.

Expected runtime semantics:

- If `copilot_enabled = false`, Copilot is `disabled` and silent.
- If `copilot_cli_enabled = false`, the CLI half is disabled and silent.
- If `copilot_vscode_user_profile_enabled = false`, the VS Code user-profile
  half is disabled and silent.
- If all configured Copilot paths are missing or unset, Copilot is
  `unavailable`.
- If some roots exist, Copilot participates only for the available
  customization_types and roots.
- A missing `copilot_cli_mcp_config_file` means no CLI MCP slots discovered; it
  must not make agents or skills unavailable.
- A missing `copilot_vscode_user_mcp_file` means no VS Code user MCP slots
  discovered; it must not make instructions, prompts, or agents unavailable.
- A newly created MCP server from another tool may create the configured MCP
  file with a root `servers` map.
- When both CLI and VS Code MCP roots are configured, first-render behavior
  should be deterministic and tested. The recommended default is to project MCP
  servers into both configured MCP files unless a per-root enable flag disables
  one.

Deliverables:

- US-11 style tests covering disabled, half-disabled, fully missing, partially
  present, and created-on-first-render cases.

## Risks and mitigations

- **Risk:** VS Code user-profile paths are profile-specific and unstable across
  platforms.
  - Mitigation: keep them explicitly configurable; do not hard-code fragile
    paths in core runtime. Installer helpers can offer best-effort discovery
    later.

- **Risk:** Copilot has overlapping local surfaces for agents
  (`~/.copilot/agents` and VS Code user profile).
  - Mitigation: support both roots, deduplicate by `pair_id` first and by
    `(customization_type, slug)` during first-boot reconciliation.

- **Risk:** `.chatmode.md` files are legacy.
  - Mitigation: parse/adopt them for compatibility, but render new output as
    `.agent.md` and document the migration.

- **Risk:** prompt files and custom agents share similar Markdown/YAML
  structures.
  - Mitigation: route by extension: `.prompt.md` means `slash_command`;
    `.agent.md`/`.chatmode.md` means `agent`.

- **Risk:** instruction files are not identical to Cursor `.mdc` rules.
  - Mitigation: map them to protocol `rules` as Markdown instruction artifacts,
    preserving frontmatter rather than synthesizing Cursor-specific rule types.

- **Risk:** MCP JSON shape differs between Copilot CLI and VS Code.
  - Mitigation: implement separate dialect wrappers and use fixture-based
    tests for both JSON files.

- **Risk:** MCP files contain literal secrets.
  - Mitigation: rely on `mcp_server_secret_policy`, default `refuse`.

- **Risk:** users expect repository `.github/` customizations to sync.
  - Mitigation: README and project description explicitly say repository scope
    is out of scope for this user-level adapter PR.

- **Risk:** extension-contributed instructions/prompts/agents/skills appear in
  VS Code but are not files in the user profile.
  - Mitigation: document extension-contributed customizations as out of scope.
