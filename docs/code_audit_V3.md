## 1. Priority findings

### 1.1 Data-loss window: delete-then-archive on mcp_server slot removal — MAJOR
`adoption/removal_propagator.py:118` · NFR-01 · US-05 AC-2 · architecture §8 I-3

`_remove_keyed_map_slot` calls `apply_slot(new_slot_text=None)`, which removes the slot from the shared keyed-map file on disk and *returns* the prior bytes; `archive.archive_text` is invoked only afterwards (lines 130–136) and **outside** the `try/except` removal-failure block. This is delete-then-archive. If `archive_text` raises (disk-full / permission `OSError`, or the path-escape `ValueError` at `archive.py:112`), the prior slot bytes are already gone and the exception is unhandled. It inverts the load-bearing invariant the project's Goal 3 / NFR-01 hinge on, contradicts US-05 AC-2 ("each survivor moved to archive, never destroyed"), and contradicts the method's own docstring. The single-file removal path (`_remove_file_artifact`) is correct because it uses the atomic `archive_move`. **Highest-value fix in the sweep; needs a regression test simulating an archive failure mid-removal.**

### 1.2 Code-vs-spec contradiction: Cursor enrolled in the agent & skill matrices — MAJOR
`tool_specs/cursor.py:79` · US-10 AC-3 · FR-05 · FR-06 · project_description Out-of-scope

The Cursor factory declares `io` and `config_dir_keys` cells for **five** types, including `agent` and `skill`. Because `AgenticToolSpec.supported_customization_types` reads `io.keys()`, this enrols Cursor in the FR-05 (agent) and FR-06 (skill) sync matrices. US-10 AC-3 states explicitly that "cursor supports `{rules, slash_command, mcp_server}` **only**", and `project_description.md` Out-of-scope records Cursor's agent/skill surfaces as SQLite-only (not file-based). Code and governance disagree: either the code has two cells it should not (a real sync defect that would project agent/skill artifacts into Cursor), or Cursor's file surface expanded and AC-3 + the Out-of-scope text are stale. **Needs adjudication before any fix.**

### 1.3 Round-trip asymmetry on Gemini CLI agent render — MAJOR
`gemini_cli_io.py:192` · NFR-06 · architecture §5.1 (round-trip stability)

`render_gemini_agent_md` injects a default `kind="local"` when the canonical carries no kind, but `parse_gemini_agent_md` only records `kind` when present in source frontmatter. For an agent that never declared a kind, `parse(render(c, prior))` yields `per_agentic_tool_only.gemini_cli.kind == "local"`, differing from the original canonical subset — violating the boundary contract `parse(render(c)) == c` and NFR-06 round-trip stability, which loop suppression depends on. Moderate confidence: may be a deliberate Gemini-required default, but the parse/render asymmetry is undocumented.

### 1.4 Asymmetric path-containment guard on the archive boundary — MAJOR
`archive.py:55` · architecture §8 I-1 · (latent, low practical risk)

`archive_dir_for` validates `pair_id` via `validate_pair_id` but interpolates the `side` (tool name) component directly into `<state_dir>/archive/<pair_id>/<side>` with no validation, and `archive_copy`/`archive_move` apply no `resolve()` + `is_relative_to()` containment guard — unlike `archive_text`, which does (lines 109–115). A `side` of `..` would escape the per-pair archive root. All current callers pass trusted registered tool names, so practical risk is low, but the boundary the project's data-preservation invariant pins to is asymmetrically guarded.

### 1.5 US-11 AC-6: unreadable root logs at the wrong level — MAJOR/MINOR
`tool_status.py:198` · US-11 AC-6

US-11 AC-6 requires an **ERROR**-level log line when a root exists but is unreadable (permission denied / `OSError`), distinct from the WARN used for a merely-missing root. The unreadable case is detected (the `iterdir()` `OSError` branch at 180–188) but `_log_status_transition` only emits `logging.info` / `logging.warning` for the `unavailable` status and never `logging.error`, so an unreadable root logs at WARN/INFO.

### 1.6 Inert `accepted_file_suffixes` on the Copilot agent cell — MAJOR
`tool_specs/copilot.py:50` · US-10 AC-1

The Copilot `agent` cell declares `accepted_file_suffixes=('.agent.md', '.chatmode.md', '.md')` to broaden discovery, but no production code reads `accepted_file_suffixes` (consumers exist only on the port dataclass and a test). Copilot agent files stored as `.chatmode.md` or `.md` would therefore not be discovered — a latent gap between declared layout intent and engine behaviour.

### 1.7 Architecture overstates secret-policy centralization (against a recent edit) — MINOR
`mcp_secret_policy.py:230`, `portable_archive.py:49` · architecture §1 Goal 6 · NFR-15

Goal 6's architectural consequence in `architecture.md` §1 states the `secret_policy` is "evaluated by the sync core … **not per adapter**." In practice `apply_mcp_secret_policy` is invoked from adapter and config sites (`mcp_server_io/parse.py`, `render.py`, `config.py`), and `portable_archive` re-implements the refuse/accept branching inline in both `export_to_zip` and `import_from_zip` rather than delegating to the shared gateway (policy logic duplicated across three sites). Behaviour satisfies NFR-15's "every egress boundary" wording, but the "not per adapter" centralization claim — added during the v0.5 objectives alignment — is stronger than the code. Either soften the wording or funnel the egress logic through the single `apply_mcp_secret_policy` gateway.
