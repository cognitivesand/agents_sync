# v0.5 implementation plan — `mcp_server` customization_type

This plan implements [US-13](stories/US-13-customization-type-expansion.md)'s
`mcp_server` slice plus FR-11 and NFR-15 from `project_requirements.md`. It
turns the protocol design in
`docs/agentic_tool_integration_protocol.md` §`mcp_server` into a buildable
PR (`feat/v0.5-mcp-server`).

The PR adds the `mcp_server` customization_type, the new
`SharedKeyedMapLayout` `file_layout`, a JSON format handler for that
layout, the `mcp_server_secret_policy` config key, and synthetic-adapter
tests covering parse / render / discovery / archive / removal-
propagation / secret enforcement. The original type PR shipped without
real adapter wiring; the current workspace also includes the follow-on
Claude Code, Codex, and OpenCode MCP adapters. Cursor, Gemini CLI, and
Copilot remain separate tool-specific PRs.

---

## Resolved design decisions for this PR

These were locked in before drafting (see chat log preceding this file):

- **D-mcp-1. File-format scope.** JSON/JSONC and TOML are registered. The
  `SharedKeyedMapLayout` core stays format-agnostic via a pluggable pair
  of callables (`deserialize(text) -> Mapping`,
  `serialize(obj) -> str`). TOML is used by Codex
  `config.toml[mcp_servers]`; YAML (Continue / Goose) comes with the
  tool-adapter PRs that need it by registering an additional format.

- **D-mcp-2. State encoding.** Add an optional
  `slot: str | None = None` field to `AgenticToolState`. When `slot` is
  set, `path` is the shared file and `slot` identifies the entry inside
  the keyed map. When `slot` is `None`, the entry is a per-file artifact
  as today. **No state-schema version bump** — the field is purely
  additive and defaults to `None` in `from_dict`.

- **D-mcp-3. Test surface.** Synthetic adapter only. Real adapters land
  in their own PRs.

- **D-mcp-4. Identity-injection key.** Top-level `pair_id` inside the
  slot, matching the rest of the codebase. Adapters add it on render and
  strip it on parse so the user's MCP client never sees it as a server
  config field.

## Phase 0 — State field addition (additive, no schema bump)

Files: `src/agents_sync/state.py`.

Changes:

- `AgenticToolState` gains `slot: str | None = None`.
- `to_dict` writes `"slot": self.slot` only when not `None` (keeps v3 JSON
  byte-identical for non-slot artifacts; avoids gratuitous churn on
  existing state files).
- `from_dict` reads `data.get("slot")`.
- Update `tests/test_state_schema_v3.py` with a roundtrip case asserting
  the field is absent in the serialised dict when `slot is None`.

Why additive without a bump: `STATE_SCHEMA_VERSION` records the
**structural** schema. Adding an optional, defaulted field that older
loaders can ignore is exactly what the existing canonical-doc pattern
does with `per_agentic_tool_extra`. The v0.4.3 cutover policy
(`load_state` returns `{}` on schema mismatch) would discard live state
unnecessarily.

Acceptance:

- All existing 197 tests pass unchanged.
- New roundtrip test asserts `from_dict(to_dict(state_with_slot))` is
  identity-preserving and `from_dict(to_dict(state_without_slot))`
  serialises without the key.

## Phase 1 — `SharedKeyedMapLayout` and the format registry

Files: `src/agents_sync/agentic_tool_spec.py`, new
`src/agents_sync/shared_keyed_map_formats.py`.

### 1.1 The layout class

```python
# In agentic_tool_spec.py, alongside SingleFileLayout / RulesFileLayout.
@dataclass(frozen=True)
class SharedKeyedMapLayout:
    """The artifact is one slot inside a shared keyed-map file."""
    shared_path_config_key: str         # e.g. "mcp_servers_file"
    map_key_path: tuple[str, ...]       # e.g. ("mcpServers",)
    key_field: str = "name"             # canonical field used as map key
    file_format: str = "json"           # one of the registered formats

    @property
    def extension(self) -> str:
        # Used by archive naming. Derived from file_format.
        return _FORMAT_REGISTRY[self.file_format].extension

    @property
    def fixed_file_name(self) -> str | None:
        return None  # Layout has no per-slot fixed name; whole file is shared.
```

The `extension` property feeds the archive filename (`<slot>.json.<ts>`).
`fixed_file_name` returns `None` because the file_layout protocol uses
that field to short-circuit per-file discovery, which does not apply here
— discovery for keyed-map layouts walks slots, not files.

### 1.2 The format registry

New module `shared_keyed_map_formats.py`:

```python
@dataclass(frozen=True)
class SharedKeyedMapFormat:
    extension: str                                          # ".json", ".toml", ".yaml"
    deserialize: Callable[[str], MutableMapping[str, Any]]  # text -> mapping
    serialize: Callable[[Mapping[str, Any]], str]           # mapping -> text

_FORMAT_REGISTRY: dict[str, SharedKeyedMapFormat] = {}

def register_format(name: str, format_handler: SharedKeyedMapFormat) -> None: ...
def get_format(name: str) -> SharedKeyedMapFormat: ...

_JSON_FORMAT = SharedKeyedMapFormat(
    extension=".json",
    deserialize=lambda text: json.loads(text or "{}"),
    serialize=lambda obj: json.dumps(obj, indent=2, sort_keys=False) + "\n",
)
register_format("json", _JSON_FORMAT)
```

Why a registry: future adapter PRs (Codex, Continue, Goose) extend the
registry with `"toml"` and `"yaml"` without touching the layout class.

### 1.3 `CustomizationTypeIO` impact

`CustomizationTypeIO` already carries `file_layout: FileLayout | None`.
Adding `SharedKeyedMapLayout` requires no signature changes on
`CustomizationTypeIO`. The existing `parse` / `render` callables operate
on **slot text** (the serialised single slot), not on the full shared
file — the keyed-map plumbing lives in discovery and rendering, not in
adapter IO.

Acceptance:

- `from agents_sync.agentic_tool_spec import SharedKeyedMapLayout` works.
- `get_format("json").serialize(get_format("json").deserialize(text))`
  roundtrips byte-for-byte on a minimal `{"mcpServers": {}}` fixture
  (modulo trailing newline).

## Phase 2 — DiscoveryWalker code path for shared-keyed-map

Files: `src/agents_sync/discovery.py`.

The walker currently has two branches: `io.storage == "single_file"` and
`io.storage == "directory_skill"`. Add a third branch for
`io.storage == "shared_keyed_map"`.

### 2.1 Per-poll behaviour

For each `(agentic_tool, customization_type)` cell whose IO is a
shared-keyed-map:

1. Resolve `shared_path = expand_path(config[layout.shared_path_config_key])`.
2. If the file does not exist, skip the cell (no slots to discover).
3. Read the file as text; `format_handler.deserialize(text)` to get the
   root mapping (where
   `format_handler = get_format(layout.file_format)`).
4. Traverse `map_key_path` into the mapping. If any segment is missing or
   not a mapping, log one structured warning per poll and skip the cell
   (the file exists but the map does not — first-boot before the tool
   created any slot).
5. For each `(slot_key, slot_value)` pair in the map:
   - Serialise the slot independently:
     `slot_text = format_handler.serialize(slot_value)`.
   - Compute artifact identity: `pair_id` is the slot's `pair_id` field
     if present, else `None` (will be minted later).
   - Push into the discovery output as a `CustomizationArtifactInfo`
     entry whose path is the shared file and whose `slot` is the slot key.

### 2.2 `_planned_adoption_targets` for keyed-map

For projection targets that are keyed-map layouts:

- Target path is the shared file (`expand_path(config[shared_path_config_key])`).
- Target slot is `target_slug(canonical[layout.key_field])`.
- Collision key is `(shared_file_resolved, slot_key)` — NOT the file
  alone. Two slots in the same file are distinct artifacts.

### 2.3 Collision detection

`path_collision_key` (in `rendering.py`) gains a slot-aware overload:

```python
def keyed_map_collision_key(shared_path: Path, slot: str) -> tuple[Path, str]:
    return (path_collision_key(shared_path), slot)
```

Two adapters writing to the same `(shared_file, slot)` are a collision
exactly as two adapters writing to the same per-file path is. The
existing `state_owner_for_path` is extended to take an optional `slot`
parameter; when set, ownership matches only entries whose `slot` field
also matches.

Acceptance:

- Discovery test with a synthetic adapter pointing at a fixture
  `mcp.json` containing 3 slots produces 3 `CustomizationArtifactInfo`
  entries with the correct slot keys.
- Discovery skip-on-missing-file test.
- Discovery skip-on-missing-map-key test (file exists, `mcpServers`
  absent).
- Collision test: two adapters declaring the same shared_path and slot
  block each other; two adapters sharing the file but with different
  slots do not collide.

## Phase 3 — Rendering for shared-keyed-map (read-modify-write)

Files: `src/agents_sync/rendering.py`, `src/agents_sync/adoption.py`.

### 3.1 The read-modify-write helper

New helper `apply_shared_keyed_map_slot`:

```python
def apply_shared_keyed_map_slot(
    shared_path: Path,
    layout: SharedKeyedMapLayout,
    slot_key: str,
    new_slot_text: str | None,    # None means delete the slot
) -> tuple[str | None, str]:
    """Read the shared file, replace/insert/delete the slot, atomically
    write it back. Returns (prior_slot_text, new_full_text)."""
```

Semantics:

- Reads the shared file. Missing file is treated as an empty mapping
  (creates the file with the map nested under `map_key_path`).
- Walks `map_key_path`, creating nested dicts as needed.
- If `new_slot_text is None`: removes the slot key. If the map becomes
  empty, leaves it empty (does NOT prune the path — preserves user's
  intent to have the section present).
- Otherwise: `format_handler.deserialize(new_slot_text)` to parse the
  slot, then assigns it under the slot key.
- `format_handler.serialize(root_mapping)` to render the merged file.
- Writes atomically via `atomic_write_text` (existing helper writes to a
  sibling `.tmp` then `os.replace`s — Windows-safe).
- Returns the prior slot text (`format_handler.serialize` of the prior
  value, or `None` if the slot did not previously exist) for archive
  use.

### 3.2 Concurrent-writer handling

Strategy: optimistic read-modify-write with detect-and-retry once.

1. Read file → compute new full text.
2. Re-read file. If contents differ from the first read, log a structured
   warning and retry the merge once.
3. If the second attempt also races, abort the write (do not corrupt the
   file). The pair is logged as "transient write conflict" and re-tried
   next poll. NFR-04 (self-healing) covers convergence.

This is the same model the existing single-file renderer uses
implicitly — `atomic_write_text` overwrites whatever is there. The
difference for keyed-map files is that we want to **preserve** sibling
slots and out-of-map keys, so we cannot just overwrite blindly.

### 3.3 `update_state_n_way` and `write_artifact_inplace`

`write_artifact_inplace` is extended (or paralleled by a sibling
`write_keyed_map_slot_inplace`) so adoption / sync / projection have a
single chokepoint. The function:

- For non-keyed-map layouts: behaves exactly as today.
- For keyed-map layouts: calls `apply_shared_keyed_map_slot`, archives
  the prior slot text under
  `archive/<pair_id>/<tool_name>/<slot_key><layout.extension>.<ISO>` via
  the existing `archive.archive_copy` (extended to accept literal text
  for the keyed-map case, not just a source file path).

`update_state_n_way` sets `ps.agentic_tools[tool].slot = slot_key`
whenever the IO is a keyed-map layout. Otherwise it leaves `slot` as
`None`.

Acceptance:

- Synthetic JSON test: an existing 3-slot file projects a new
  4th-slot adoption without touching the 3 existing slots'
  bytes (modulo format-normalisation; JSON allows reformatting). Use
  `format_handler.serialize(format_handler.deserialize(prior_full_text))`
  as the byte-equality baseline.
- Sibling-preservation test: out-of-map keys in the file (e.g. `theme`
  in Gemini's `settings.json`) are preserved byte-for-byte across a
  write.
- Concurrent-writer test (manually injected): the helper detects a mid-
  read change and retries; a second mid-read change aborts cleanly.

## Phase 4 — Removal propagation for keyed-map slots

Files: `src/agents_sync/removal.py` (or wherever the existing per-pair
removal handler lives — confirm during implementation), `adoption.py`.

When a slot is removed on one tool (the source-of-truth tool's slot
disappears), the existing removal logic deletes the per-file artifact
from every other tool. For keyed-map projection targets the operation is
to remove the slot key from each other tool's shared file:

1. For each surviving tool whose state entry has a `slot`:
   - Archive the prior slot value (via the rendering helper above with
     `new_slot_text=None`).
   - Call `apply_shared_keyed_map_slot(..., new_slot_text=None)` to
     delete the key.
2. For each surviving tool whose state entry has no `slot` (per-file
   artifact): existing behaviour (archive then unlink).

The "shared file goes away while we have slots tracked" case (e.g. user
deleted `~/.cursor/mcp.json` entirely) is treated as removal of every
slot the state knows about for that tool. Same logic as a per-file
artifact going missing — the existing reconciler already covers this.

Acceptance:

- Removal test: a synthetic-adapter pair whose source tool's slot is
  deleted causes the projection slot in the other tool's shared file to
  be removed; sibling slots remain.
- "Shared file deleted" test: removes the file on tool A; the engine
  archives every previously-tracked slot from tool B's mirror.

## Phase 5 — `mcp_server` customization_type and the secret policy

Files: new `src/agents_sync/mcp_server_io.py`, new
`src/agents_sync/mcp_secret_policy.py`,
`src/agents_sync/agentic_tool_spec.py`, `src/agents_sync/config.py`.

### 5.1 The IO module

`mcp_server_io.py` is the canonical parser / renderer for one MCP slot,
parametrised by the per-adapter dialect (transport-name aliases, field
spellings). The v0.5 type PR ships only the canonical implementation;
adapter PRs supply their dialect tables.

Functions:

- `parse_mcp_server_json(slot_text, prior_canonical, *, agentic_tool_name,
   artifact_path, artifact_root, dialect=...) -> canonical`
- `render_mcp_server_json(canonical, prior_text, *, agentic_tool_name,
   dialect=...) -> slot_text`
- `extract_pair_id_from_mcp_server_json(slot_text) -> str | None`

Canonical schema (matches protocol §`mcp_server` Canonical document
fields):

- `pair_id` (string, required) — engine-managed.
- `kind = "mcp_server"`.
- `name` (string, required) — canonical identity; equals the slot key.
- `transport` (`"stdio" | "http" | "sse" | "streamable-http"`,
  required).
- `command`, `args`, `env`, `cwd`, `timeout` — when `transport == "stdio"`.
- `url`, `headers`, `auth` — when transport is HTTP-family.
- `disabled`, `always_allow` — optional, common.
- `secret_redactions` — present only when policy was `redact`.
- `per_agentic_tool_extra` — opaque pass-through, same shape as existing
  artifacts.

### 5.2 Secret policy module

`mcp_secret_policy.py` is the single source of truth for the
detection heuristic and the three policies. It is called by
`parse_mcp_server_json` and `render_mcp_server_json` — NOT by every
adapter, so policy behaviour is uniform.

Heuristic (matches protocol §Secret-redaction policy):

- Flag any string literal in `env`, `headers["Authorization"]`,
  `headers["X-API-Key"]`, `auth.client_secret`, or any field name
  matching `(?i)(api[_-]?key|token|secret|password)`, whose value is not
  already an `${env:VAR}` reference (regex
  `^\$\{env:[A-Z_][A-Z0-9_]*\}$`).

Policy behaviour:

- `"refuse"`: raise `McpSecretLeakError` with a structured payload (slot
  path, offending field paths). The caller (parse / render) propagates
  it as a US-03 AC-10 parse failure — the artifact is NOT adopted and
  the prior on-disk state is left untouched.
- `"redact"`: replace each offending literal with
  `${env:AGENTS_SYNC_REDACTED_<n>}` and record
  `{field_path, original_env_var}` in `canonical.secret_redactions`.
  `original_env_var` is `null` unless a sibling field (e.g. Codex's
  `env_key`) names the variable.
- `"permissive"`: leave literals untouched; emit one structured warning
  per artifact per poll naming the field paths. Returns
  `policy_warnings: list[str]` alongside the canonical so the engine
  can surface them.

### 5.3 Config wiring

`src/agents_sync/config.py`:

- New top-level key `mcp_server_secret_policy: str` with default
  `"refuse"`.
- `REQUIRED_DIR_KEYS` is unchanged (the key is a string policy, not a
  directory).
- `validate_config` validates the value is one of the three allowed
  strings.
- CLI flag `--mcp-server-secret-policy {refuse,redact,permissive}` added
  to `build_parser`.

Acceptance:

- Policy test matrix (synthetic): for each of the three policies, the
  heuristic detects literals in each documented field and behaves per
  spec.
- `secret_redactions` roundtrip test: redacting then re-parsing yields
  the placeholder in canonical and the recorded redaction entries.
- Negative test: invalid policy value at startup raises `ConfigError`
  with the policy name in the message.

## Phase 6 — Synthetic adapter and end-to-end integration tests

Files: new `tests/test_mcp_server_io.py`, new
`tests/test_mcp_server_sync.py`.

### 6.1 Synthetic adapter

A `_mcp_server_spec(tool_name, shared_path_key)` factory in the test
file builds an `AgenticToolSpec` with:

- `supported_customization_types = {"mcp_server"}`
- `config_dir_keys = {"mcp_server": shared_path_key}`
- IO whose `file_layout = SharedKeyedMapLayout(shared_path_key,
  ("mcpServers",), file_format="json")`
- `parse / render` from `mcp_server_io.parse_mcp_server_json /
  render_mcp_server_json` with an identity dialect.

### 6.2 Tests

The PR ships at least these integration tests (mirrors
`test_rules_sync.py` / `test_slash_command_sync.py`):

1. `test_mcp_adopt_between_synthetic_adapters` — both tools start with a
   3-slot `mcpServers` map; one tool gains a new slot; the engine adopts
   and projects to the other tool's map preserving siblings.
2. `test_mcp_pair_id_injected_into_slot` — the slot in the source tool
   gains a top-level `pair_id` after first sync; second sync sees it as
   the identity.
3. `test_mcp_archive_is_per_slot` — modifying one slot archives only
   that slot's prior text; sibling slot bytes are not archived even
   though the file as a whole is rewritten.
4. `test_mcp_remove_slot_removes_in_other_tool` — deleting a slot in the
   source map removes it from the projection map and archives the prior
   value.
5. `test_mcp_secret_policy_refuse_blocks_adoption` — a slot with a
   literal `API_KEY` value fails parse under `refuse`; no projection
   target is touched.
6. `test_mcp_secret_policy_redact_rewrites` — same input under `redact`
   produces a `${env:AGENTS_SYNC_REDACTED_1}` placeholder and a
   `secret_redactions` entry in canonical.
7. `test_mcp_secret_policy_permissive_warns` — same input under
   `permissive` propagates verbatim and logs one structured warning.
8. `test_mcp_sibling_keys_outside_map_preserved` — out-of-map top-level
   keys (e.g. `theme: "dark"`) survive a slot write byte-for-byte
   (modulo format normalisation tracked by the JSON format's stable
   output).
9. `test_mcp_first_boot_reconciliation_by_slot_key` — two tools each
   start with a slot named `github` and no `pair_id`; the engine
   reconciles them as a single pair using the most-recent shared-file
   mtime, same logic as today's per-file reconciliation.
10. `test_mcp_two_adapters_same_shared_file_different_slots_no_collision`
    — `path_collision_key` returns `(shared_file, slot)`, not just
    `shared_file`, so two slots in the same file are independent
    artifacts.
11. `test_mcp_two_adapters_same_shared_file_same_slot_collide` — the
    negative of #10: same shared_file + same slot blocks both pairs.
