# v0.5 implementation plan - Gemini CLI adapter

This plan implements the Gemini CLI slice of `v0.5`, on the dedicated branch
`feat/v0.5-gemini-cli`.

Gemini CLI is a real adapter for local, file-backed agent customizations. It
combines the existing `customization_types` with the v0.5 types that have a
clear Gemini CLI file surface:

- `agent`, via `~/.gemini/agents/*.md` custom subagents
- `skill`, via `~/.gemini/skills/<name>/SKILL.md`
- `rules`, via `~/.gemini/GEMINI.md` persistent context
- `slash_command`, via `~/.gemini/commands/**/*.toml`
- `mcp_server`, via `~/.gemini/settings.json[mcpServers]`

Gemini CLI also has important customization surfaces that are intentionally not
modeled in this PR:

- extension packages under `~/.gemini/extensions/<name>/`
- hooks under extension `hooks/hooks.json`
- Policy Engine TOML files
- themes
- built-in subagent overrides in `settings.json`
- workspace-scoped `.gemini/` files

The most important difference from the Cursor adapter is that Gemini CLI does
not use Cursor-style `.mdc` rules. Gemini CLI persistent instructions are
provided through `GEMINI.md` context files, so the adapter maps the user-level
`~/.gemini/GEMINI.md` file to the existing `rules` customization_type. Gemini
CLI safety/permission behavior is controlled through the Policy Engine; that is
a separate surface and should not be forced into `rules`.

## Why Gemini CLI

Gemini CLI is a strong v0.5 adapter candidate because it has first-class,
documented file-backed surfaces for persistent context, subagents, Agent
Skills, custom slash commands, and MCP servers.

It is also a useful cross-tool interoperability target:

- Gemini CLI subagents are Markdown files with YAML frontmatter.
- Gemini CLI skills use the open `SKILL.md` directory shape.
- Gemini CLI persistent rules/context are Markdown in `GEMINI.md`.
- Gemini CLI custom commands are TOML files under a command directory.
- Gemini CLI MCP servers are configured in a shared JSON settings file under
  the `mcpServers` key.

Adding Gemini CLI proves that the registry model can support a modern terminal
agent without treating every AI tool as an IDE. The adapter should stay focused
on documented user-level files and avoid project-specific or extension-package
state unless a later feature explicitly supports those scopes.

## Scope

### In scope

- Register a new `gemini_cli` agentic_tool.
- Support Gemini CLI custom subagents from:
  `~/.gemini/agents/*.md`.
- Support Gemini CLI Agent Skills from:
  `~/.gemini/skills/<name>/SKILL.md`.
- Support Gemini CLI user rules from:
  `~/.gemini/GEMINI.md`.
- Support Gemini CLI user slash commands from:
  `~/.gemini/commands/**/*.toml`.
- Support Gemini CLI MCP server slots from:
  `~/.gemini/settings.json` under root key `mcpServers`.
- Per-OS defaults:

  | OS | Gemini CLI home | Agents | Skills | Rules | Commands | MCP/settings |
  |---|---|---|---|---|---|---|
  | Linux | `~/.gemini/` | `~/.gemini/agents/` | `~/.gemini/skills/` | `~/.gemini/GEMINI.md` | `~/.gemini/commands/` | `~/.gemini/settings.json` |
  | macOS | `~/.gemini/` | `~/.gemini/agents/` | `~/.gemini/skills/` | `~/.gemini/GEMINI.md` | `~/.gemini/commands/` | `~/.gemini/settings.json` |
  | Windows | `%USERPROFILE%\.gemini\` | `%USERPROFILE%\.gemini\agents\` | `%USERPROFILE%\.gemini\skills\` | `%USERPROFILE%\.gemini\GEMINI.md` | `%USERPROFILE%\.gemini\commands\` | `%USERPROFILE%\.gemini\settings.json` |

- `gemini_cli_enabled` config key, default `true`.
- Runtime `unavailable` status when a configured Gemini CLI root is missing.
- Secret handling for MCP servers through the global
  `mcp_server_secret_policy`.
- Unit and integration tests covering all five supported types.
- README, protocol notes, installer sample config, and requirements matrix
  updates.

### Out of scope

- Project-scoped files:
  - `<project>/.gemini/agents/`
  - `<project>/.gemini/skills/`
  - `<project>/.agents/skills/`
  - `<project>/.gemini/commands/`
  - `<project>/.gemini/settings.json`
  - `<project>/GEMINI.md`
- User-level `~/.agents/skills/` alias. It can be considered in a follow-up
  once alias precedence is modeled explicitly.
- Gemini CLI extensions under `~/.gemini/extensions/<name>/`.
- Extension `gemini-extension.json` packaging.
- Gemini CLI hooks.
- Gemini CLI Policy Engine TOML files.
- Gemini CLI themes.
- Key-level sync of non-MCP `settings.json` settings.
- Built-in subagent overrides under `settings.json[agents.overrides]`.
- Remote Agent2Agent subagents.
- Gemini Code Assist IDE-specific state.

## Goals

1. A Gemini CLI custom subagent syncs to every other available tool that
   supports `agent`, and agents from other tools render into Gemini CLI's
   subagent Markdown format.
2. A Gemini CLI Agent Skill syncs to every other available tool that supports
   `skill`, including auxiliary files next to `SKILL.md`.
3. A Gemini CLI `GEMINI.md` rules artifact syncs to every other available tool
   that supports `rules`, and rules from other tools render into Gemini CLI's
   user-level `GEMINI.md`.
4. A Gemini CLI slash command syncs to every other available tool that supports
   `slash_command`, preserving prompt content and supported TOML metadata.
5. A Gemini CLI MCP server slot syncs through `mcp_server` without touching
   sibling keys in `~/.gemini/settings.json`.
6. Gemini CLI can be enabled on a machine without Gemini CLI installed; missing
   roots make the tool `unavailable`, not fatal.
7. Gemini CLI's unsupported surfaces are documented clearly so users do not
   expect `agents_sync` to move Policy Engine files, hooks, themes, or
   extension packages.

## Non-goals

- Auto-detect Gemini CLI installation.
- Sync project-scoped Gemini CLI customizations.
- Translate Policy Engine TOML files into `rules`.
- Translate extension packages into individual agent/skill/command artifacts.
- Persist identity outside the managed artifacts.
- Add new sync-engine behavior.

## Baseline findings

- Gemini CLI is an open-source terminal agent that uses built-in tools and
  local or remote MCP servers.
- Gemini CLI custom subagents are Markdown files with YAML frontmatter and a
  Markdown body used as the agent system prompt.
- User-level Gemini CLI subagents are expected under
  `~/.gemini/agents/*.md`; project-level `.gemini/agents/*.md` is intentionally
  out of scope.
- Gemini CLI skills are `SKILL.md` directories. User-level skills are discovered
  from `~/.gemini/skills/` and the `~/.agents/skills/` alias. This PR uses
  `~/.gemini/skills/` as the default managed path and leaves alias handling for
  a future follow-up.
- Gemini CLI custom commands are TOML files with a required `prompt` field and
  optional metadata such as `description`.
- Gemini CLI command names are derived from the path under `commands/`;
  subdirectories become colon namespaces.
- Gemini CLI MCP servers live in `settings.json[mcpServers]`.
- Gemini CLI extensions can bundle prompts/context, MCP servers, custom
  commands, hooks, subagents, skills, policies, and themes, but installed
  extension packages are out of scope for this adapter PR.
- Gemini CLI `GEMINI.md` files provide persistent instructional context. This
  PR maps the user-level `~/.gemini/GEMINI.md` file to `rules`; project-level
  and extension-level context files remain out of scope.
- Gemini CLI subagents are still a fast-moving surface, so release validation
  must include a manual Gemini CLI smoke test.

## Resolved design decisions for this PR

- **D-gem-1. User scope only.** The adapter reads and writes only the
  user-level `~/.gemini/` home. Project-scope files remain owned by the user's
  repositories.
- **D-gem-2. Supported types.** Gemini CLI declares exactly
  `{agent, skill, rules, slash_command, mcp_server}`.
- **D-gem-3. Rules mapping.** Gemini CLI maps the user-level
  `~/.gemini/GEMINI.md` persistent context file to the existing `rules`
  customization_type. Project-level and extension-level `GEMINI.md` files
  remain out of scope. Gemini CLI Policy Engine TOML files are not equivalent
  to Markdown rules and are deferred until the protocol has an explicit
  `policy` customization_type.
- **D-gem-3a. Rules identity.** Gemini CLI rules use the global rules helper
  with a fixed filename `GEMINI.md`, canonical name `global`, and `pair_id`
  stored in YAML frontmatter.
- **D-gem-4. Agent identity.** Gemini CLI subagents are Markdown files under
  `~/.gemini/agents/*.md`. The adapter stores `pair_id` in YAML frontmatter,
  preserves canonical fields such as `name`, `description`, `model`, `tools`,
  and body, and stores Gemini-only fields under
  `per_agentic_tool_only["gemini_cli"]`.
- **D-gem-5. Skill identity.** Gemini CLI skills use the open `SKILL.md`
  directory shape under `~/.gemini/skills/<name>/SKILL.md`. The adapter shares
  the same open-spec helper used by other skill-capable tools wherever
  possible.
- **D-gem-6. Slash-command identity.** Gemini CLI commands are TOML files. The
  adapter stores `pair_id` as a TOML field, preferably
  `agents_sync_pair_id = "<uuid>"`, unless the v0.5 slash-command type
  standardizes a different identity carrier.
- **D-gem-7. Slash-command content.** The canonical `body` maps to the TOML
  `prompt` field. The TOML `description` field maps to canonical
  `description`.
- **D-gem-8. MCP identity.** Gemini CLI MCP slots use the shared v0.5
  `mcp_server_io` convention: top-level `pair_id` inside the server slot, with
  the slot rendered through `settings.json[mcpServers]`.
- **D-gem-9. Recursive command names.** `~/.gemini/commands/git/commit.toml`
  maps to canonical command name `git:commit`, matching Gemini CLI's command
  namespacing behavior.
- **D-gem-10. Skill nesting.** Gemini CLI skills render one folder deep:
  `~/.gemini/skills/<slug>/SKILL.md`. Nested category folders are not used.
- **D-gem-11. Secret policy.** Gemini CLI MCP secrets are handled only by
  `mcp_server_secret_policy`; the Gemini CLI adapter does not invent a
  separate policy.
- **D-gem-12. Settings preservation.** Rendering MCP servers into
  `~/.gemini/settings.json` must preserve unrelated settings keys, including
  UI, model, context, tools, security, skills, and hooks config.

## Workstreams

### 1) Gemini CLI IO module

Target module: new `src/agents_sync/gemini_cli_io.py`.

Implement these helpers:

- `extract_pair_id_from_gemini_agent_md(text) -> str | None`
- `parse_gemini_agent_md(text, prior_canonical, *, artifact_path, artifact_root)`
- `render_gemini_agent_md(canonical, prior_text=None) -> str`
- `extract_pair_id_from_gemini_skill_md(text) -> str | None`
- `parse_gemini_skill_md(text, prior_canonical, *, artifact_path, artifact_root)`
- `render_gemini_skill_md(canonical, prior_text=None) -> str`
- Gemini rules wrappers around shared `rules_io` for fixed file
  `~/.gemini/GEMINI.md`.
- `extract_pair_id_from_gemini_command_toml(text) -> str | None`
- `parse_gemini_command_toml(text, prior_canonical, *, artifact_path, artifact_root)`
- `render_gemini_command_toml(canonical, prior_text=None) -> str`
- Gemini MCP wrappers around shared `mcp_server_io` with Gemini CLI's
  `settings.json[mcpServers]` dialect.

Agent behavior:

- Parse Gemini CLI subagents as Markdown with YAML frontmatter plus body.
- Require or preserve frontmatter fields:
  - `name`
  - `description`
  - `kind`
  - `tools`
  - `mcpServers`
  - `model`
  - `temperature`
  - `max_turns`
- Derive `name` from the filename stem only when frontmatter is missing a name.
- Round-trip common fields: `name`, `description`, `model`, `tools`, and body.
- Preserve Gemini-only fields such as `kind`, `temperature`, `max_turns`,
  `mcpServers`, and run/model knobs under
  `per_agentic_tool_only["gemini_cli"]`.
- Preserve unknown frontmatter in `per_agentic_tool_extra["gemini_cli"]`.
- Render to `~/.gemini/agents/<target_slug>.md`.
- Keep the Markdown body byte-stable when possible.

Skill behavior:

- Reuse the shared open-spec `SKILL.md` parser/renderer where possible.
- Render to `~/.gemini/skills/<target_slug>/SKILL.md`.
- Preserve auxiliary files exactly like existing skill adapters.
- Round-trip `name`, `description`, `license`, `compatibility`, `metadata`,
  and Gemini-specific optional fields.
- Do not render nested skill category directories.
- Do not write to `~/.agents/skills/` in this PR.

Rules behavior:

- Reuse the shared v0.5 Markdown `rules_io` helper wherever possible.
- Render one user-level file: `~/.gemini/GEMINI.md`.
- Use canonical rules name `global`, matching other single global context
  files such as `CLAUDE.md` and `AGENTS.md`.
- Store `pair_id` in YAML frontmatter.
- Preserve Markdown body content as the Gemini persistent instructions.
- Preserve supported rules metadata such as `description`, `globs`, `applyTo`,
  and `alwaysApply` where the shared rules helper supports them.
- Preserve unknown frontmatter in `per_agentic_tool_extra["gemini_cli"]`.
- Do not read or write project-level `<project>/GEMINI.md`.
- Do not read or write extension-level `~/.gemini/extensions/<name>/GEMINI.md`.
- Do not parse Gemini CLI Policy Engine TOML files as rules.

Slash-command behavior:

- Parse TOML files.
- Map `prompt` to canonical `body`.
- Map `description` to canonical `description`.
- Preserve unknown TOML keys in `per_agentic_tool_extra["gemini_cli"]`.
- Treat command names as path-derived:
  - `commands/test.toml` -> `test`
  - `commands/git/commit.toml` -> `git:commit`
- Render to `~/.gemini/commands/<target_slug>.toml`.
- Render namespaced commands by replacing `:` with path separators.
- Preserve TOML formatting where practical, but prioritize semantic stability
  over byte-for-byte TOML formatting.
- Do not use Markdown identity comments. Gemini CLI commands are TOML.

MCP behavior:

- Parse Gemini CLI `settings.json[mcpServers]`.
- Preserve unrelated `settings.json` keys exactly.
- Parse supported Gemini CLI transports:
  - `command` means stdio;
  - `url` means SSE endpoint;
  - `httpUrl` means streamable HTTP.
- Preserve server fields such as `args`, `env`, `cwd`, `timeout`, `trust`,
  `headers`, `oauth`, and `excludeTools` where present.
- Preserve unknown server fields in `per_agentic_tool_extra["gemini_cli"]`.
- Let shared secret-policy code handle `env`, `headers`, and auth material.

Deliverables:

- New Gemini CLI IO module.
- Focused IO unit tests for agents, skills, rules, commands, and MCP slot
  dialect.

### 2) Agentic-tool spec registration

Target module: `src/agents_sync/agentic_tool_spec.py`.

Add `_build_gemini_cli_spec()`:

```python
return AgenticToolSpec(
    name="gemini_cli",
    config_dir_keys={
        "agent": "gemini_cli_agents_dir",
        "skill": "gemini_cli_skills_dir",
        "rules": "gemini_cli_rules_dir",
        "slash_command": "gemini_cli_commands_dir",
        "mcp_server": "gemini_cli_settings_file",
    },
    io={
        "agent": CustomizationTypeIO(
            parse=parse_gemini_agent_md,
            render=render_gemini_agent_md,
            extract_pair_id=extract_pair_id_from_gemini_agent_md,
            storage="single_file",
            file_suffix=".md",
        ),
        "skill": CustomizationTypeIO(
            parse=parse_gemini_skill_md,
            render=render_gemini_skill_md,
            extract_pair_id=extract_pair_id_from_gemini_skill_md,
            storage="directory_skill",
            file_suffix="",
        ),
        "rules": _global_rules_io("gemini_cli", "GEMINI.md"),
        "slash_command": CustomizationTypeIO(
            parse=parse_gemini_command_toml,
            render=render_gemini_command_toml,
            extract_pair_id=extract_pair_id_from_gemini_command_toml,
            file_layout=SlashCommandFileLayout(extension=".toml"),
            slugify_name=slash_command_slug,
            recursive=True,
        ),
        "mcp_server": CustomizationTypeIO(
            parse=parse_gemini_mcp_server_json,
            render=render_gemini_mcp_server_json,
            extract_pair_id=extract_pair_id_from_gemini_mcp_server_json,
            file_layout=SharedKeyedMapLayout(
                shared_path_config_key="gemini_cli_settings_file",
                map_key_path=("mcpServers",),
                file_format="json",
            ),
        ),
    },
    disable_config_key="gemini_cli_enabled",
)
```

Then add `"gemini_cli": _build_gemini_cli_spec()` to
`default_agentic_tools()` in alphabetical order.

### 4) Runtime availability behavior

Target modules: no sync-engine changes expected.

Expected runtime semantics:

- If `gemini_cli_enabled = false`, Gemini CLI is `disabled` and silent.
- If all configured Gemini CLI paths are missing, Gemini CLI is `unavailable`.
- If some roots exist, Gemini CLI participates only for the available
  customization_types.
- A missing `GEMINI.md` means no Gemini rules artifact is discovered; it must
  not make agents, skills, slash commands, or MCP slots unavailable when their
  roots exist.
- A missing `gemini_cli_settings_file` means no Gemini MCP slots discovered; it
  must not make agents, skills, rules, or slash commands unavailable.
- A newly created rules artifact from another tool creates
  `~/.gemini/GEMINI.md` if the Gemini CLI root exists.
- A newly created MCP server from another tool creates `~/.gemini/settings.json`
  with a root `mcpServers` map, preserving any existing settings if the file
  already exists.

Deliverables:

- US-11 style tests covering disabled, fully missing, partially present, and
  created-on-first-render cases.

## Risks and mitigations

- **Risk:** Gemini CLI subagents are still evolving.
  - Mitigation: preserve unknown frontmatter in
    `per_agentic_tool_extra["gemini_cli"]`, pin tests to real Gemini CLI
    fixtures, and include manual CLI validation before release.

- **Risk:** Gemini CLI commands use TOML, while other tools use Markdown.
  - Mitigation: map canonical `body` to the TOML `prompt` field, preserve
    `description`, and keep unknown TOML keys in tool-specific extra data.

- **Risk:** users expect Cursor-style `.mdc` rule files for Gemini CLI.
  - Mitigation: explicitly document that Gemini CLI `rules` are backed by the
    user-level `~/.gemini/GEMINI.md` file, while Cursor remains backed by
    Cursor `.mdc` files.

- **Risk:** users expect Gemini CLI Policy Engine TOML files to sync as
  `rules`.
  - Mitigation: document Policy Engine files as out of scope for this PR and
    reserve them for a future explicit `policy` customization_type.

- **Risk:** users expect installed extensions to sync.
  - Mitigation: keep this adapter focused on user-level file surfaces and
    document extension packages as out of scope.

- **Risk:** Gemini CLI skills are discoverable from both `~/.gemini/skills/`
  and `~/.agents/skills/`, with alias precedence.
  - Mitigation: write only to `~/.gemini/skills/` in this PR and document
    alias support as a possible follow-up.

- **Risk:** MCP files contain literal secrets.
  - Mitigation: rely on `mcp_server_secret_policy`, default `refuse`.

- **Risk:** MCP projection rewrites unrelated Gemini CLI settings.
  - Mitigation: shared-keyed-map rendering must preserve unrelated
    `settings.json` keys and sibling `mcpServers` entries.
