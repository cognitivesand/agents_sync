## Phase 1 — SEC-B-01: Path-traversal closure (release-blocker)

**Goal:** make `archive_text` provably unable to write outside `<state_dir>/archive/<pair_id>/<side>/`, regardless of attacker-supplied MCP slot keys.
**Risk if skipped:** a hostile MCP slot name like `"../../../tmp/pwned"` — delivered via tampered portable archive OR via direct edit of `.claude.json` / `~/.codex/config.toml` / `opencode.json` — escapes the archive containment boundary and writes prior-slot bytes (which under `mcp_server_secret_policy=permissive` may include Authorization headers or env credentials) to an attacker-chosen path.

### Scope

| # | Change |
|---|---|
| 1.1 | **Introduce `archive._sanitize_slot_component(slot_name: str) -> str`** in `archive.py`. Strips `/`, `\`, `..`, NUL, control chars (`\x00`–`\x1f`); applies the existing `state.slugify` semantics; caps at 128 chars; falls back to `"slot"` when empty. Unit-tested directly. |
| 1.2 | **`archive_text` calls the sanitizer** on `slot_name` before path construction. After joining, asserts `target.resolve().is_relative_to(target_dir.resolve())`; raises `ValueError("archive target escapes per-pair directory")` if not. The assertion is belt-and-braces — the sanitizer is the load-bearing fix. |
| 1.3 | **Regression test** `tests/test_archive_text_security.py::test_archive_text_rejects_traversal_in_slot_name` — feeds `../../../tmp/pwned`, `foo/bar`, `\\..\\evil`, `\x00poison`, ` ` (whitespace-only), `""` (empty), and a 500-char name; asserts the resulting path is inside the archive dir and the slot-name component matches a strict allowlist. |
| 1.4 | **End-to-end regression** `tests/test_archive_text_security.py::test_traversal_in_mcp_slot_key_does_not_escape_archive_dir` — wires the full discovery → adoption → archive_text pipeline with an MCP slot named `"../../../evil"` in a synthetic `mcp.json`; asserts the archive ends up inside `archive/<pair_id>/<side>/`. |

### Commit

```
fix(sec): sanitize MCP slot name before archive_text path join (SEC-B-01)

Raw MCP slot keys from .claude.json/.codex/config.toml/opencode.json were
interpolated into archive target filenames unchanged, letting a slot named
'../../../tmp/pwned' escape <state_dir>/archive/<pair_id>/<side>/.

Closes audit finding: SEC-B-01 (HIGH)
```

### Open decision

**D1: sanitization strictness.** Should the slug allow `_`, `.`, `-`, or only `[a-z0-9-]`? The audit recommendation says "reuse `state.slugify` or stricter." `state.slugify` already returns `[a-z0-9_-]+`. Recommendation: **reuse `state.slugify` verbatim** to keep one source of truth; the value of slot names in the archive is reproducibility, not legibility. If the user wants the original key it remains in the canonical document. Confirm before implementing.

## Phase 2 — SEC-A-01: Generic `secret_policy` + per-artifact egress filters (MAJOR)

**Goal:** deliver US-12 AC-12…16 (export/import secret handling) and the structural reshape the 2026-05-22 requirements rewrite implies — generic `secret_policy` config key, binary `secrets_refused` / `secrets_accepted` values, per-artifact filtering at both egress boundaries.
**Risk if skipped:** the `secrets_refused` contract has no teeth at the egress points; stale canonicals from before policy tightening leak verbatim through `export`; a `secrets_accepted` source-host export imposes its stance on every receiving host with no recourse.
**Depends on:** none (independent of Phase 1).

**Governance status:** governance edits already landed in commits `1051a0a`, `de2d572`, `aa55e24` — `project_requirements.md` reflects the new policy values and the cross-cutting NFR-15, `US-12` carries AC-12…16, and `US-13` carries the parse-time generic ACs. Phase 2 is now pure code + test work.

### Scope

| # | Change |
|---|---|
| 2.1 | **Config-key rename in code.** `mcp_server_secret_policy` → `secret_policy` (generic, future-proof per the rewritten NFR-15). One-release compatibility shim in `config.merged_config`: accept either spelling; when the old spelling is present, emit one structured DEPRECATION-WARNING log line at startup naming the file and the new spelling. CLI flag `--mcp-server-secret-policy` is renamed to `--secret-policy`; the old flag is kept as a deprecated alias for one release with the same warning. |
| 2.2 | **Policy value collapse.** Drop `redact` mode entirely from `mcp_secret_policy.apply_mcp_secret_policy`. Two outcomes remain: under `secrets_refused`, raise `McpSecretLeakError`; under `secrets_accepted`, return the canonical unchanged with a structured WARNING. Compatibility shim in `config.merged_config`: accept the old spellings `refuse`/`permissive` and map them to `secrets_refused`/`secrets_accepted` respectively; map the old `redact` spelling to `secrets_refused` with a DEPRECATION-WARNING noting the safer default (redact mode is gone — see below). Existing canonicals whose `secret_redactions` blocks already contain `${env:AGENTS_SYNC_REDACTED_<n>}` placeholders continue to round-trip transparently because the placeholders are env-references and the heuristic never flagged env-references. |
| 2.3 | **Export-time filter (US-12 AC-12, AC-13, AC-14).** `portable_archive.export_to_zip` walks every canonical, calls `mcp_secret_policy.find_secret_literals(canonical, artifact)`, and applies the active policy per-artifact: under `secrets_refused`, **skip** the canonical and log one structured WARNING per skipped artifact naming `customization_artifact_id` + field path (no abort); under `secrets_accepted`, include verbatim and log one summary WARNING listing the affected `customization_artifact_id`s. Manifest gains a single boolean `contains_secret_literals` (NO `secret_pair_ids` list — per FR-14 rewrite that decision was made noise). |
| 2.4 | **Import-time filter (US-12 AC-15, AC-16).** `portable_archive.import_from_zip` walks every candidate canonical, calls `find_secret_literals(canonical, artifact)`, and applies the receiver's active policy per-artifact: under `secrets_refused`, skip with one structured WARNING per skipped artifact (clean artifacts still import per AC-5 / AC-6); under `secrets_accepted`, import all and log one summary WARNING. The receiver's policy ALWAYS overrides whatever the source-host policy was — AC-15 makes this defence-in-depth explicit. |
| 2.5 | **Per-artifact reporting on `ExportReport` / `ImportReport`.** Add `skipped_secret_artifacts: list[str]` (pair_ids that were filtered out) to both dataclasses so the CLI can print a single-line summary and tests can assert without scraping log records. |
| 2.6 | **Tests `tests/test_portable_archive_secret_egress.py`** — one test per AC: (a) AC-12 secrets_refused no-literals → full export, manifest false; (b) AC-13 secrets_refused with a stale literal-bearing canonical → clean canonicals shipped, the one with literals skipped, manifest false, WARNING logged with structured event; (c) AC-14 secrets_accepted with a literal-bearing canonical → verbatim export, manifest true, WARNING logged; (d) AC-15 import under secrets_refused of an export with literals → clean canonicals imported, secret-bearing skipped with WARNING; (e) AC-16 import under secrets_accepted → all imported with WARNING. Plus three compat-shim tests: (f) old config key `mcp_server_secret_policy` still accepted, DEPRECATION-WARNING logged; (g) old value `refuse` maps to `secrets_refused`; (h) old value `redact` maps to `secrets_refused` with DEPRECATION-WARNING. |
| 2.7 | **README** updates the secret-policy section to document the new values, the deprecation path, the egress behaviour, and the cross-host trust model (receiver's policy always wins). |

### Design note — the `--allow-secrets` flag is GONE

The earlier Phase 2 draft proposed a `--allow-secrets` CLI opt-in flag that overrode `secrets_refused` at export time. That was wrong: the policy IS the opt-in. If the operator wants to export literal secrets, they set `secret_policy = secrets_accepted`; the export then ships secrets verbatim with a manifest declaration and a warning. The defence against "permissive set for debugging and forgotten" is now at the **receiver** — a target host with `secret_policy = secrets_refused` skips secret-bearing canonicals on import (AC-15), so even a sloppy producer can't impose secrets on a careful consumer.

### Commit

```
feat(sec): generic secret_policy + per-artifact egress filters (SEC-A-01)

Implements US-12 AC-12..16 + the secret-policy rename driven by the
2026-05-22 requirements rewrite (commit aa55e24).

- Rename mcp_server_secret_policy -> secret_policy in code, config, CLI.
- Collapse refuse/redact/permissive -> secrets_refused/secrets_accepted.
- Remove redact mode; existing redaction placeholders in canonicals
  continue to round-trip as env-references.
- Export filters per-artifact: under secrets_refused, skip + WARN;
  under secrets_accepted, include verbatim + WARN. Manifest carries
  a single contains_secret_literals boolean.
- Import applies the receiver's policy symmetrically.
- One-release compatibility shim accepts the old config-key and value
  spellings with DEPRECATION-WARNING logs.

Closes audit finding: SEC-A-01 (MAJOR)
```

### Open decisions

**D2: code rename blast radius.** `mcp_server_secret_policy` appears in source, tests, the README, the install scripts (`.env` defaults), and config-validation messages. The rename + shim is intrusive but bounded. Recommendation: **land the rename atomically in this phase** rather than spreading it across follow-ups; a half-renamed codebase is worse than either end state.

**D5: deprecation horizon.** How long does the compat shim accept the old spellings? Recommendation: **one minor release (v0.5 ships both; v0.6 ships only the new spelling)**. The DEPRECATION-WARNING log line gives users a window to migrate; a hard cut at the next minor release keeps the shim from becoming permanent.

**D6: stale-canonical migration command.** The audit's SEC-A-04 finding (stale canonicals retain literal secrets after policy tightening) is partially addressed by AC-13 (export skips them) but the canonicals themselves remain on disk. Recommendation: **defer to a separate follow-up** — a `agents-sync canonical-rescan` subcommand is its own design exercise and isn't a release-blocker. Phase 2 stays scoped to the egress filters.

## Phase 3 — SEC-C-01 + SEC-C-02: Parser DoS bounds (MAJOR, bundled)

**Goal:** prevent the long-running daemon from being DoS'd by either oversized input files or pathological YAML alias graphs. Both findings concern parser boundaries and share one design seam.
**Risk if skipped:** a 2 GB hostile `mcp.json` OOMs the daemon every poll; a sub-100 KB YAML bomb hangs `split_frontmatter`. Both kill the daemon without recovery.

### Scope

| # | Change |
|---|---|
| 3.1 | **New module `src/agents_sync/parser_bounds.py`** owning `MAX_PARSE_BYTES = 16 * 1024 * 1024` (16 MB), `MAX_FRONTMATTER_BYTES = 256 * 1024` (256 KB), `MAX_YAML_ANCHORS = 100`, `MAX_YAML_ALIAS_RESOLUTIONS = 10_000`. Single source of truth. |
| 3.2 | **Size-cap at every parser entry point**: `formats/json_format.deserialize`, `formats/toml_format.deserialize`, `mcp_server_io/_slot_codec.loads_slot`, `shared_keyed_map_io.read_slots`, `shared_keyed_map_io._read_root_and_node`, `canonical.load_canonical`, `state.load_state`. Each checks `len(text) > MAX_PARSE_BYTES` and raises `ParserBoundsExceeded("input exceeds MAX_PARSE_BYTES (16 MB)")` before parsing. Where the call site has the path, prefer `path.stat().st_size > MAX_PARSE_BYTES` before reading. |
| 3.3 | **Frontmatter size-cap** in `markdown_yaml_metadata_block.split_frontmatter` — `FRONTMATTER_RE.match(text[:MAX_FRONTMATTER_BYTES])` so the regex linear scan is bounded regardless of body size. |
| 3.4 | **YAML alias cap** via a small `BoundedComposer` subclass of `ruamel.yaml.composer.Composer` that counts `compose_node` invocations and raises `ParserBoundsExceeded` after `MAX_YAML_ALIAS_RESOLUTIONS`. Wired into `markdown_yaml_metadata_block.make_yaml()`. |
| 3.5 | **Exception type**: `ParserBoundsExceeded` subclasses `AdapterParseError` so existing callers (discovery walker's broad `except Exception`, quarantine path in `load_canonical`/`load_state`) handle it gracefully. The daemon's per-poll failure path already records the affected pair_id in `SyncResult.failed`; this slots in without code changes there. |
| 3.6 | **Tests** `tests/test_parser_bounds.py`: (a) 17 MB JSON input rejected with `ParserBoundsExceeded`; (b) 257 KB frontmatter prefix rejected; (c) classical billion-laughs YAML rejected within ~50 ms; (d) a benign 5 MB JSON parsed successfully; (e) alias count of 9_999 succeeds, 10_001 fails; (f) corrupt input that triggers the cap is quarantined per the existing contract, not silently swallowed. |

### Commit

```
feat(sec): bound parser inputs against DoS (SEC-C-01, SEC-C-02)

Introduce parser_bounds.py owning MAX_PARSE_BYTES (16 MB),
MAX_FRONTMATTER_BYTES (256 KB), MAX_YAML_ALIAS_RESOLUTIONS (10_000),
and ParserBoundsExceeded (AdapterParseError subclass). All parser entry
points enforce the caps before they hand text to ruamel/json/tomllib.
A BoundedComposer subclass counts alias resolutions and aborts pathological
YAML graphs.

Closes audit findings: SEC-C-01 (MAJOR), SEC-C-02 (MAJOR)
```

### Open decision

**D3: cap values.** 16 MB / 256 KB / 10_000 aliases are conservative defaults from the audit recommendation. Two-week production observation may show legitimate inputs near these bounds (large MCP server configs in shared dotfiles). Recommendation: **start with these values, expose them as constants** (not config), and revisit once production data is available. Configurability adds a CLI surface and a documented contract; defer until the constants prove too tight.

## Phase 4 — TQ-01: Fast-path discipline (MAJOR, mechanical)

**Goal:** make `pytest -m 'not integration'` actually deselect the integration-class tests, so CLAUDE.md §7.1's fast/full pytest split is enforced.
**Risk if skipped:** the fast-path promise is decoration only. Every iteration cycle pays the full integration cost; developers will turn the test suite into background work and lose the inner-loop discipline.
**Depends on:** none.

### Scope

| # | Change |
|---|---|
| 4.1 | **Apply `pytestmark = pytest.mark.integration`** at module level in: `tests/test_e2e_sync.py`, `tests/test_first_boot_reconciliation.py`, `tests/test_v0_4_1_matrix.py`, `tests/test_migrate_v0_4_e2e.py`, `tests/test_mcp_real_adapters.py`, `tests/test_rules_real_adapters.py`, `tests/test_slash_command_real_adapters.py`, `tests/test_v0_4_1_matrix.py`, `tests/test_shared_keyed_map_*.py`. |
| 4.2 | **Apply `pytest.mark.slow`** to individual tests whose wall-clock exceeds ~2 s, per CLAUDE.md §7.1. Identified by running `uv run pytest --durations=20` once and tagging the offenders. |
| 4.3 | **CI check.** Add a tiny `tests/test_marker_discipline.py::test_integration_layer_carries_marker` that imports the integration modules and asserts each has a module-level `pytestmark` covering `integration`. So a future test file added to that layer without a marker fails CI. |
| 4.4 | **README** updates the "How to run tests" section to point at `pytest -m 'not integration'` for the fast path and `pytest` for the full path. |

### Commit

```
test(disc): apply slow/integration markers to enforce fast/full pytest split (TQ-01)

The slow/integration markers were registered in pyproject.toml but applied
to zero modules. `pytest -m 'not integration'` deselected nothing.
Apply pytestmark = pytest.mark.integration to every integration-class
module and add a marker-discipline guard.

Closes audit finding: TQ-01 (MAJOR)
```

### Open decision

**D4: marker discipline guard.** Item 4.3 is a meta-test — it asserts a property of the test suite itself. Alternative: rely on the README + reviewer discipline. Recommendation: **keep the guard.** Once the discipline is in place, a single forgotten marker on a new integration test silently re-erodes the policy; a 10-line meta-test is the cheap insurance.
