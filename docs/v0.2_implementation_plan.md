# agents_sync v0.2 — Implementation Plan

This is the engineering plan for the v0.2 rewrite. It is the bridge between `project_description.md` / `project_requirements.md` / `docs/stories/` and the actual code change. No code is being written yet; this document is for review.

## Goals of this rewrite

- Replace the one-way Claude → Codex translator with a bidirectional daemon.
- Introduce a per-pair canonical JSON document as the lossless intermediate; the "review metadata" JSON blob in v0.1 outputs is removed.
- Inject `pair_id` UUIDs into both sides for identity that survives rename.
- Implement archive-on-overwrite for data preservation.
- Implement last-`mtime` conflict resolution.
- Drop the one-shot CLI mode; the daemon is the only execution mode.

## Non-goals (this phase)

- Automatic migration from v0.1 (no install base to migrate; the user removes the v0.1 install manually).
- inotify / fsevents (polling at the configured interval is sufficient).
- Multi-user, multi-host, project-scoped agents.
- File-level merge inside skill folders (pair-level conflict only).

## Module layout

The single ~410-line `src/claude_codex_sync/cli.py` is split into focused modules under `src/agents_sync/`:

```
src/agents_sync/
  __init__.py         version
  __main__.py         python -m agents_sync entry
  cli.py              argument parsing, daemon wiring
  config.py           TOML config loading, path expansion, defaults
  state.py            state.json index, canonical/<pair_id>.json store
  archive.py          archive paths, ISO timestamps, archive-on-overwrite
  canonical.py        canonical schema, schema versioning, validation
  claude_io.py        Claude .md parse + render via ruamel.yaml
  codex_io.py         Codex .toml parse + render
  sync.py             per-pair sync algorithm, adoption, conflict resolution
  daemon.py           polling loop, signal handling, transient-exception recovery
```

Each module has a narrow surface and is independently testable.

## Canonical schema

Stored at `~/.local/state/agents-sync/canonical/<pair_id>.json`. Shape (informal):

```jsonc
{
  "schema_version": 1,
  "pair_id": "<UUIDv4>",
  "kind": "agent" | "skill",
  "name": "<string>",
  "description": "<string>",
  "body": "<markdown body or developer_instructions content>",
  "model": "<string>|null",
  "effort": "low|medium|high|null",
  "tools": ["<string>", ...],
  "disallowed_tools": ["<string>", ...],
  "permission_mode": "<string>|null",

  // Side-specific structured fields with no semantic mapping
  "claude_only": { "hooks": ..., "mcp_servers": ... },
  "codex_only":  { "sandbox_mode": "<string>|null" },

  // Unstructured passthrough buckets — guarantee no field is silently dropped
  "claude_extra": { ... },
  "codex_extra":  { ... }
}
```

Design notes:

- `*_extra` buckets carry any frontmatter / TOML keys we don't model explicitly.
- `*_only` carry structured fields that exist only on one side.
- `schema_version` is the migration knob; bumping it archives the prior canonical before rewriting.

## Renderer / parser contracts

Each side exposes the same two functions:

```python
def render(canonical: dict) -> str: ...
def parse(text: str, prior_canonical: dict | None) -> dict: ...
```

Required invariants:

- **NFR-06**: `parse(render(c), c) == c` for every valid canonical `c`.
- **Determinism**: `render(c)` is byte-deterministic across runs.
- `parse` takes the prior canonical so that, when the user has edited only mapped fields, the unmapped passthrough buckets survive.

The v0.1 "Converted Claude-specific metadata for manual review" JSON blob is removed entirely. Fidelity lives in the canonical; rendered files are clean.

## State layout

```
~/.local/state/agents-sync/
  state.json                                    # thin index per pair_id
  canonical/<pair_id>.json                      # one canonical doc per pair
  archive/<pair_id>/<side>/<filename>.<ISO>     # preserved prior bytes
```

`state.json` shape:

```jsonc
{
  "<pair_id>": {
    "kind": "agent" | "skill",
    "claude_path":         "/abs/path/foo.md",
    "codex_path":          "/abs/path/foo.toml",
    "claude_last_seen":    "<sha256>",
    "codex_last_seen":     "<sha256>",
    "claude_last_written": "<sha256>",
    "codex_last_written":  "<sha256>"
  },
  ...
}
```

The decision rule per pair on each poll:

| current claude digest | current codex digest | action |
|---|---|---|
| `== last_written` (claude) | `== last_written` (codex) | noop |
| `!= last_written`           | `== last_written`           | claude → canonical → codex |
| `== last_written`           | `!= last_written`           | codex → canonical → claude |
| `!= last_written`           | `!= last_written`           | mtime-wins; archive loser; resolve |
| missing                     | present                     | propagate removal: archive codex, drop pair |
| present                     | missing                     | symmetric |

## Archive policy

Archive **on**:

- First-time `pair_id` injection into a previously untouched user file.
- Conflict tiebreaker — the loser's prior bytes.
- Rename-induced filename change on the counterpart side (prior filename's content).
- Schema bump that rewrites the canonical (prior canonical).
- Adoption of foreign artifacts (prior bytes before pair_id injection).

Archive **off**:

- Routine retranslation overwrites where the prior content equals a render of the current canonical (NFR-07: bounded archive growth).
- Internal staging dirs (`.<name>.tmp`, `.<name>.old`).

Timestamp format: ISO 8601 UTC with `:` replaced by `-` (e.g., `2026-05-06T16-35-01Z`).

## Sync algorithm (per poll cycle)

```
1. enumerate Claude side, Codex side
2. for each file: parse pair_id (or flag as foreign)
3. group into { pair_id: {claude_path?, codex_path?} } merging with state
4. for each pair: apply the decision-table action above
5. for each foreign artifact: adopt (archive original → mint pair_id → inject →
   build canonical → render counterpart → record state)
6. flush state.json atomically
```

Per-pair errors are caught, logged at WARN, and other pairs continue (FR-02).

## Atomic-swap recipes

Single file (Claude `.md` or Codex `.toml`):

1. Render bytes to `.<name>.tmp` adjacent to target.
2. `rename(2)` `.tmp` → target.

Skill folder:

1. Stage full target tree at `.<name>.tmp` (copytree from source side, then overwrite SKILL.md with rendered version from canonical).
2. If target exists: `rename` target → `.<name>.old`.
3. `rename` `.tmp` → target.
4. Archive `.old` (only if it contained user-authored bytes); otherwise `rmtree` internally — exempt from never-rm rule because it's a throwaway staging dir.

## Configuration

`~/.config/agents-sync/config.toml`:

```toml
[agents-sync]
poll_interval_seconds = 2.0
claude_agents_dir = "~/.claude/agents"
claude_skills_dir = "~/.claude/skills"
codex_agents_dir  = "~/.codex/agents"
codex_skills_dir  = "~/.agents/skills"
state_dir         = "~/.local/state/agents-sync"
```

Every key is overridable via CLI flag (e.g., `--poll-interval-seconds 5.0`).

## Logging strategy

- **INFO**: startup, shutdown, per-pair operations (adoption, sync direction, rename, archive).
- **WARN**: conflicts (with mtime + winner + archive paths), transient exceptions, slug collisions, mtime-tie tiebreaker.
- **ERROR**: configuration errors, parse failures, archive write failures.
- **DEBUG**: per-poll digest diffs.

Format: one line per event, structured `key=value` pairs.

## Dependency changes

Add:

- `ruamel.yaml >= 0.18` — roundtrip-preserving YAML for Claude `.md` frontmatter (NFR formatting preservation, already implicit in US-04 AC-4).
- `tomli-w >= 1.0` — proper TOML writer for Codex `.toml`; replaces v0.1's hand-rolled `toml_string`.

Drop:

- `PyYAML` (replaced by ruamel.yaml).

Keep:

- `tomllib` (stdlib, Python 3.12+).

## Phasing

Five landings, each independently buildable, testable, and committable:

1. **Rename + module split** (no behaviour change)
   - `claude_codex_sync` package → `agents_sync`.
   - Console script `claude-codex-sync` → `agents-sync`.
   - `install.sh` paths and systemd unit name.
   - `cli.py` content split across the modules listed above; logic preserved.
   - Sanity check: existing one-way sync still works.

2. **Canonical + state refactor (still one-way)**
   - Inject `pair_id` into both sides on first encounter; archive originals.
   - Build canonical from current Claude side; render Codex from canonical.
   - Replace path-keyed state with `pair_id`-keyed state; track per-side digests.
   - Round-trip property tests on render/parse pairs.

3. **Reverse direction**
   - Add `parse_codex(text, prior_canonical) -> canonical'`.
   - Decision-table sync algorithm wired in.
   - Last-`mtime` conflict resolution.

4. **Daemon-only execution**
   - Remove `--once` and the corresponding argparse branch.
   - Default invocation runs the daemon.
   - `install.sh --service` becomes the canonical install path.

5. **Polish**
   - Foreign artifact adoption tested end-to-end.
   - Skill folder atomic swap under tests (NFR-03).
   - Archive growth bounding tests (NFR-07).
   - Long-run resource stability harness (NFR-05, NFR-08).

Phase 1 is the risky step (rename + restructure with no semantic change). Tag the v0.2-pre commit before Phase 2 so we can bisect if something regresses.

## Test plan

**Property tests (per side):**

- `parse(render(c), c) == c` over generated canonicals (NFR-06).
- `render(c)` is byte-deterministic across runs.
- A poll with no input changes produces zero writes (NFR-05).

**End-to-end fixtures (temp Claude / Codex dirs):**

- Edit Claude side → Codex updates within 2 polls.
- Edit Codex side → Claude updates within 2 polls.
- Rename Claude file on disk → state path updates, no re-render.
- Change `name` field → counterpart filename changes, old filename archived.
- Foreign Codex artifact appears → adopted, `pair_id` injected, Claude counterpart created.
- Both sides edited inside one poll window → mtime winner takes, loser archived.
- Daemon killed mid-write → restart converges (atomic-write invariant).
- Two daemons briefly overlap → atomic writes preserve correctness.

**Long-run smoke test:**

- Run daemon for N hours with no input changes; assert zero writes, zero archive entries, stable RSS (NFR-05, NFR-08).

## Tool rename mechanics

Done in Phase 1:

- `pyproject.toml`: `name = "agents_sync"`, version bumped to `0.2.0`, `[project.scripts] agents-sync = "agents_sync.cli:main"`.
- `src/claude_codex_sync/` → `src/agents_sync/`.
- `install.sh`: `APP_NAME="agents-sync"`; all derived paths follow.
- systemd unit: `agents-sync.service`.
- `README.md`: update commands and the table of source/target paths.

The v0.1 install at `~/.config/claude-codex-sync/` and `~/.local/state/claude-codex-sync/` is **not** auto-migrated. On first start, if either path exists, the daemon logs a clear ERROR pointing at them and exits with the configuration-error exit code. The user removes them manually (or moves them to the v0.1 archive of their own choosing). For a single-user developer tool with no production install base, this is acceptable.

## Risks & open questions

- **ruamel.yaml roundtrip fidelity**: real-world YAML edge cases (anchors, multi-line strings with specific quoting) may still reformat slightly. Mitigation: property tests with diverse samples; document known limits.
- **TOML writer determinism**: `tomli-w` may reorder keys vs. user-authored ordering. Codex `.toml` files we own end-to-end — but adopted foreign artifacts won't preserve original ordering. Acceptable tradeoff or worth a separate investigation?
- **Rename + name change in the same poll**: pair_id resolves identity, but the counterpart-filename change adds a second operation. Test explicitly.
- **Skill folder partial-edit conflict**: user edits SKILL.md on Claude side and `asset.txt` on Codex side simultaneously — the design treats this as a single pair-level conflict, with mtime-wins on the whole tree. Per-file resolution is a future enhancement.
- **Pair-id slug collisions during adoption**: two foreign Claude agents whose `name` slugifies to the same value; US-03 AC-5 says reject. Concrete handling: leave both untouched, log a structured ERROR naming both paths, do not adopt either.
- **Configuration file location**: `~/.config/agents-sync/` is opinionated. If the user wants `XDG_CONFIG_HOME` respected, that's a small addition; flag if you want it.

## Sequencing

Suggested order of work after plan approval:

1. Tag the current `main` as `v0.1-final-docs`.
2. Land Phase 1 (rename + split) in one PR; re-verify v0.1 behaviour.
3. Land Phase 2 (canonical + state refactor) in a second PR; one-way still.
4. Land Phase 3 (reverse direction + conflict resolution).
5. Land Phase 4 (daemon-only).
6. Land Phase 5 (polish + long-run tests).

Each landing leaves the daemon usable; no half-finished states on `main`.
