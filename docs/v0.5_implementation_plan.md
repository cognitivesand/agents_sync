# v0.5 implementation plan — Cursor, Gemini CLI, GitHub Copilot

This plan adds **three new agentic_tools** (Cursor, Gemini CLI / Antigravity, GitHub Copilot) and **three new customization_types** (`rules`, `slash_command`, `mcp_server`) to `agents_sync`. The design choices are anchored in `docs/agentic_frameworks_user_data_library.md` and `docs/agentic_frameworks_compatibility_analysis.md`.

Without the new customization_types, Cursor has no syncable surface (no agent files, no `SKILL.md` skills); Gemini CLI is partial; and GitHub Copilot is the CLI half only. The new types are therefore not optional for meaningful v0.5 coverage.

## Scope summary

In scope:

- New `customization_type = rules` — one Markdown file per rule, optional YAML frontmatter. Covers `CLAUDE.md`, `GEMINI.md`, `AGENTS.md`, `*.instructions.md`, `.cursor/rules/*.mdc`, and the equivalent user-scope files across the three frameworks.
- New `customization_type = slash_command` — one file per command, Markdown body with optional frontmatter; TOML variant required for Gemini CLI's `commands/*.toml`.
- New `customization_type = mcp_server` — one MCP server per canonical artifact, projected to whatever shared-keyed-map file the framework uses (`~/.cursor/mcp.json`, `~/.gemini/settings.json[mcpServers]`, `~/.copilot/mcp-config.json` and `<vscode-user>/mcp.json`). The shared-keyed-map projection is a new `file_layout` value — see §Protocol additions.
- New agentic_tool modules: `cursor.py`, `gemini_cli.py`, `copilot.py`. (Antigravity remains a separate adapter sharing skills support with Gemini CLI; or merged — open question §Open questions Q1.)
- Memory: handled as `rules` with two new canonical fields, `provenance: "user" | "agent"` and `private: true|false`. No `memory` customization_type. See `docs/agentic_frameworks_compatibility_analysis.md` §2.8.

Out of scope for v0.5:

- `hooks` customization_type (deferred to v0.6 — event-taxonomy projection is non-trivial).
- `mode` customization_type (deferred — only Roo / Kilo demand it, neither in v0.5).
- `settings.json` key-level extraction (deferred — different abstraction).
- Cursor's SQLite-only surfaces (User Rules pane, in-app Custom Modes, Memories, Notepads). Explicitly out of scope; the Cursor adapter documents the gap.
- AI Assistant Prompt Library and Junie/AI Assistant XML state (irrelevant to v0.5; would be a JetBrains adapter, not in scope here).
- Background Agents secrets, GitHub Copilot coding-agent cloud configuration (server-side, not on disk).

Constraints:

- v0.4 protocol surface preserved. `AgenticToolSpec` and `CustomizationTypeIO` remain the only entry points for new tools. NFR-11 (extensibility) is the load-bearing requirement.
- No edits to the sync engine, conflict resolution, adoption, reconciliation, or removal-propagation logic. The new customization_types ride the existing algorithms.
- Project-scope (`<project>/.cursor/`, `<project>/.gemini/`, `<project>/.github/`) remains out of scope — only user-level customizations sync, per the project description.

## Release strategy

Per scope decision (one PR per customization_type, then one PR per framework adapter), v0.5 ships as six follow-on PRs after this plan/protocol PR merges:

1. **This PR (`feat/v0.5-plan`)** — plan, US-13, FR additions, protocol additions for the three new customization_types, glossary updates. No code changes; non-governance edits applied directly, governance edits paused for user approval.
2. `feat/v0.5-rules` — `rules` customization_type with file_layout, canonical schema fields (`provenance`, `private`), engine wiring, and integration tests against a synthetic adapter. No real adapters yet.
3. `feat/v0.5-slash-command` — `slash_command` customization_type, including the TOML variant. Same scope shape as the rules PR.
4. `feat/v0.5-mcp-server` — `mcp_server` customization_type, **including the new shared-keyed-map `file_layout`**. The largest of the three type PRs.
5. `feat/v0.5-cursor` — Cursor adapter exercising all three new types. First real consumer.
6. `feat/v0.5-gemini-cli` — Gemini CLI / Antigravity adapter. Adds TOML slash-command parser; extends or replaces the existing `antigravity.py` (see Open question Q1).
7. `feat/v0.5-copilot` — GitHub Copilot adapter covering the CLI half (`~/.copilot/`) and the VS Code user-profile half (`*.instructions.md`, `*.prompt.md`, `*.chatmode.md`/`*.agent.md`, user `mcp.json`).

Each adapter PR may land an FR-XX entry in `docs/project_requirements.md` recording its concrete matrix, mirroring FR-05 / FR-06.

## Protocol additions

The `agentic_tool_integration_protocol.md` gets new sections specifying the three new `customization_type` values. Each section follows the existing `agent` / `skill` cadence:

1. **`rules`** customization_type:
   - `file_layout = SingleFileLayout(extension=".md")` — same shape as `agent` today.
   - Identity field: filename stem (slug).
   - Frontmatter is optional; canonical document carries the body verbatim and the following declared fields when present in frontmatter: `description`, `globs` (string or list), `applyTo` (string), `alwaysApply` (bool), `trigger` (enum, framework-specific — stays in `per_agentic_tool_only`).
   - New canonical fields `provenance: "user" | "agent"` (default `"user"`) and `private: true|false` (default `false`). The engine respects `private: true` by excluding the artifact end-to-end. Set by the adapter at parse time based on the source path.
   - Project-scope `AGENTS.md` and `CLAUDE.md` remain out of v0.5 sync scope (user-level only).
   - Cursor's `.mdc` extension is a per-adapter override of `extension`; the protocol allows the adapter to declare `extension=".mdc"`.

2. **`slash_command`** customization_type:
   - `file_layout = SingleFileLayout(extension=".md" | ".toml")` — Markdown is the default, Gemini CLI sets TOML.
   - Identity field: filename stem (or filename path under `commands/`, with subdirectories namespacing to `<dir>:<stem>` per Claude/Codex/Gemini convention).
   - Canonical frontmatter fields: `description`, `argument-hint`, `allowed-tools`, `model`, `agent`/`mode`. Body verbatim. Body interpolation grammars (`$ARGUMENTS`, `!`-shell, `@`-file) are stored as-is; per-tool parse/render does not normalise them.

3. **`mcp_server`** customization_type:
   - `file_layout = SharedKeyedMapLayout(shared_path, map_key_path, key_field="name")` — **new file_layout value**. The adapter declares one or more shared files (e.g. `~/.cursor/mcp.json` user + `<project>/.cursor/mcp.json` project), the JSON path to the keyed map inside (`mcpServers`), and the key field that becomes the identity.
   - Identity field: the map key (server name).
   - Canonical fields: `name`, `transport` (`stdio` | `http` | `sse` | `streamable-http`), plus transport-tagged fields (`command`, `args`, `env`, `cwd`, `url`, `headers`, `auth`). Per-tool aliasing of transport name (`local`/`remote` from opencode, `streamableHttp` from Cline, etc.) goes in `per_agentic_tool_only`.
   - Secret handling: a canonical field `secret_redaction_policy` records, per artifact, which fields were redacted on export. Default policy is **strict** — refuse to sync artifacts containing literal high-entropy strings in `env`, `headers`, `auth.*`, falling back to `${env:VAR}` indirection (open question Q2 below).
   - The sync engine treats the *shared file* as the archive granularity: on write, the prior shared file is archived once, then mutated atomically. Concurrent edits to two MCP servers in the same shared file in a single poll merge cleanly via existing self-healing-poll logic (NFR-04).

## State schema impact

Adding new customization_types does NOT require a state-schema bump. The existing `state.customization_artifacts[<id>].agentic_tools[<name>]` map is already polymorphic over customization_types. Per-pair `last_modified` (v3, shipped in v0.4.3) covers conflict resolution for the new types unchanged.

The new `provenance` and `private` fields live in the **canonical document**, not in state. Canonical schema is versioned independently and admits unknown fields (existing pattern from `per_agentic_tool_extra`).

## Resolved design decisions

Four design decisions were resolved with the user before code work begins:

- **D1. Antigravity adapter stays separate from Gemini CLI.** v0.4.1's `antigravity.py` (skills-only, pointing at `~/.gemini/antigravity/skills/`) is preserved unchanged. A new `gemini_cli.py` is added covering agents, `rules` (`GEMINI.md`), `slash_command` (TOML, `commands/*.toml`), and `mcp_server` (`settings.json[mcpServers]`). The two adapters coexist; their shared physical paths are governed by D3 below.

- **D2. MCP-server secret-redaction policy is a config setting, default refuse.** A new top-level config key, `mcp_server_secret_policy`, accepts `"refuse"` (default), `"redact"`, or `"permissive"`:
  - `"refuse"` — the daemon will not write an `mcp_server` artifact whose canonical document contains a literal value matching the secret-detection heuristics on any of `env`, `headers`, `auth.*`. The user must use `${env:VAR}` indirection. Fail-closed.
  - `"redact"` — literals are replaced with a placeholder (`${env:AGENTS_SYNC_REDACTED_<n>}`); the original variable hint is stored in `per_agentic_tool_only.<tool>.secret_redactions` so the user can re-resolve manually on each host.
  - `"permissive"` — literals propagate as-is; a structured warning is logged per artifact per poll.
  NFR-15 codifies this; US-13's acceptance criteria pin the heuristics and the error/warning shape.

- **D3. Shared physical paths use the path-ownership model.** When two registered adapters look at the same physical directory for the same customization_type, exactly one adapter is the **owner**; others are **aliases**. Concrete v0.5 case: `antigravity` owns `~/.gemini/antigravity/skills/` for `skill`; `gemini_cli` does NOT declare `skill` in its `supported_customization_types`. The canonical records the artifact under `antigravity` only.
  Protocol surface: no new field needed for v0.5 — the path-ownership rule is enforced by the adapter declarations themselves (the alias adapter omits the customization_type from `supported_customization_types`). A future protocol enhancement may add an explicit `aliases_of` field on `config_roots` if the alias adapter *needs* read access (e.g. for diagnostics). v0.5 does not require this.

- **D4. Cursor's `.mdc` extension is a per-adapter file_layout choice.** No protocol change. The existing `AgentFileLayout(extension=".<ext>")` mechanism already lets each adapter declare its own extension per customization_type (Codex declares `.toml`, others `.md`). The new `RulesFileLayout` class added in v0.5 carries the same per-adapter `extension` parameter; Cursor's `AGENTIC_TOOL` declares `RulesFileLayout(extension=".mdc")`, every other adapter declares `RulesFileLayout(extension=".md")`. Killed as an architectural question.
