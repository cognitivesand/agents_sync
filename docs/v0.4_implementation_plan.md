# agents_sync v0.4 — Implementation Plan

This document defines the engineering plan for `v0.4`: adding **Google Antigravity** as a third agentic_tool in the sync, **for skills only**, while preserving all existing Claude Code <-> Codex behavior unchanged.

## Why v0.4

Google Antigravity (agentic IDE, GA Nov 2025) has adopted the same open `SKILL.md` spec that Claude Code and Codex skills use. Antigravity skills are plain files on disk at well-known paths, with no cloud lock-in. A growing community library (1,400+ skills) already cross-distributes the same `SKILL.md` files across Claude Code, Codex, Cursor, Gemini CLI, and Antigravity. Users running all three tools currently maintain three copies by hand.

`v0.4` extends `agents_sync` so any change to a skill on any of the three agentic_tools propagates to the other two, with the same loss-less, archive-before-overwrite, `customization_artifact_id`-anchored guarantees the project already provides for two-agentic_tool skill and agent sync.

## Scope

### In scope

- Add **Antigravity** as a third agentic_tool in the sync.
- Antigravity participates in **skills only**. Agents are unchanged.
- User-level skills only (`~/.gemini/antigravity/skills/<name>/SKILL.md`). Workspace-level skills remain out of scope (parity with current treatment of Claude project skills).
- Three-way bidirectional sync with last-mtime-wins conflict resolution generalized to N agentic_tools.
- Adoption of new Antigravity skills (no `customization_artifact_id`) symmetric with existing adoption.
- Removal propagation: removing a skill on any one agentic_tool archives the other two.
- Linux, macOS, and Windows path defaults for Antigravity (existing macOS support remains best-effort).
- State schema cut over to v2 (per-agentic-tool dicts). No v1 reader: the project is pre-1.0 with two known users (jmirodg, gabi) who regenerate state on first v0.4 boot.

### Out of scope

- Antigravity **agents**. Antigravity has no stable, user-level, per-agent file format analogous to `~/.claude/agents/*.md` or `~/.codex/agents/*.toml`. The `agents.md` workspace-level multi-persona prompt and the experimental "Custom Agents and Plugins" flag are explicitly excluded. Re-evaluate when Google ships a stable per-agent file spec.
- Workspace-scoped (project-level) skills on any agentic_tool.
- Gemini CLI, Cursor, Windsurf, or other consumers of the open SKILL.md spec. Antigravity is the only third agentic_tool added in v0.4.
- Migration of arbitrary unknown artifact metadata beyond the existing passthrough-bag mechanism.

## Goals

1. Editing a skill on any of the three agentic_tools propagates to the other two within at most two polling intervals.
2. Existing Claude<->Codex behavior (both agents and skills) is unchanged when Antigravity is disabled or its directory is empty.
3. No user-authored content is destroyed: every overwrite or removal on any of the three agentic_tools archives the prior bytes under the existing layout.
4. Adding Antigravity to an already-managed Claude+Codex customization_artifact requires no manual intervention.
5. Antigravity-specific artifact metadata fields (`license`, `compatibility`, `metadata`, `allowed-tools`) round-trip losslessly. Claude-specific fields (`model`, `tools`, `hooks`, `mcpServers`) are preserved through Antigravity storage as opaque passthrough.

## Non-goals

- Replace polling with platform watchers.
- Auto-detect Antigravity install location. The Antigravity skills root is configurable; if the configured root does not exist, Antigravity becomes `unavailable` per US-11 (the daemon stays alive, logs the transition once, and resumes when the root reappears).
- Resolve simultaneous concurrent writes from a third-party tool to the same skill mid-poll.

## Baseline findings (from current code)

- Agentic_tools are hard-coded throughout. `PairInfo` has `claude` / `codex` SideInfo fields. `PairState` has `claude_path` / `codex_path` / `claude_last_*` / `codex_last_*`. `Syncer` has parallel `_add_claude_*` / `_add_codex_*` / `_sync_*_to_*` / `_propagate_*_removal` / `_adopt_from_*` methods.
- Canonical has `claude_only` / `codex_only` plus `claude_extra` / `codex_extra` for per-agentic_tool passthrough.
- Two-agentic_tool mtime conflict resolution lives in a single `_resolve_conflict` method.
- Adoption is symmetric per agentic_tool; target-collision detection is centralized in `_block_target_collisions`.
- All atomic-write, archive, and `stage_skill_dir` primitives are already agentic_tool-agnostic.

**Implication:** to add a third agentic_tool cleanly we must generalize the agentic_tool dimension from "two named fields" to "a registry of N named agentic_tools, each declaring which customization_artifacts it handles." Hand-coding a third parallel set of `_add_antigravity_*` methods would triple the parallel surface area and make N-way conflict resolution unmaintainable.

## v0.4 workstreams

### 1) Agentic-tool registry refactor (pre-Antigravity, no behavior change)

Generalize the agentic_tool dimension so adding a third agentic_tool becomes a config + backend-module change rather than a sync-algorithm rewrite. **The N-way sync algorithm (discovery / adoption / conflict resolution / removal propagation / §5.5 first-boot reconciliation) is implemented in this workstream, operating on a participating-agentic_tool set of length 2 (`claude`, `codex`).** Phase 4 then becomes config and registry wiring, not algorithm work. Done before any Antigravity code is added, so each refactor PR is reviewable in isolation against the existing two-agentic_tool test suite.

Concretely:

- Introduce `AgenticToolSpec` descriptor: `name` (e.g. `"claude"`, `"codex"`, `"antigravity"`), `supported_customization_types` (subset of `{"agent", "skill"}`), and the IO module that implements `parse` / `render` / `extract_customization_artifact_id` (as `CustomizationTypeIO` triples) for each customization_type it supports.
- Rename `PairInfo` to `CustomizationArtifactInfo` and replace its `claude` / `codex` fields with `agentic_tools: dict[str, AgenticToolInfo]` (where `AgenticToolInfo` is the renamed `SideInfo`).
- Rename `PairState` to `CustomizationArtifactState` and replace its `claude_path` / `codex_path` / `claude_last_*` / `codex_last_*` with `agentic_tools: dict[str, AgenticToolState]` (renamed from `SideState`), where `AgenticToolState` holds `path`, `last_seen`, `last_written`.
- Replace `claude_only` / `codex_only` / `claude_extra` / `codex_extra` in canonical with `per_agentic_tool_only: dict[str, dict]` and `per_agentic_tool_extra: dict[str, dict]`.
- Collapse `_add_claude_*` / `_add_codex_*` into `_add_agentic_tool_artifact(agentic_tool_name, customization_artifact, ...)`.
- Collapse `_sync_claude_to_codex` / `_sync_codex_to_claude` / `_resolve_conflict` into `_sync_agentic_tool_to_others(winner_agentic_tool, ...)` and a generic `_resolve_conflict_n_way(...)` that picks `max(mtime)` over participating agentic_tools.
- Collapse `_adopt_from_claude` / `_adopt_from_codex` into `_adopt_from_agentic_tool(agentic_tool_name, ...)`.
- Collapse `_propagate_*_removal` into `_propagate_removal(missing_agentic_tool, present_agentic_tools, ...)`.

State schema:

- Bump canonical `SCHEMA_VERSION` from `1` to `2`.
- `state.json` becomes a versioned envelope: `{"schema_version": 2, "customization_artifacts": {<customization_artifact_id>: {...}}}`.
- Canonical replaces `claude_only` / `claude_extra` / `codex_only` / `codex_extra` with `per_agentic_tool_only` / `per_agentic_tool_extra` outright.
- **No v1 reader.** The project has two users (jmirodg, gabi); their existing state files are regenerated on first v0.4 boot. v2 is the only on-disk shape from now on.

Test coverage: all existing tests pass after the refactor against v2-only fixtures.

Target modules: `sync.py`, `state.py`, `canonical.py`, `claude_io.py`, `codex_io.py`, `config.py`.

Deliverables:

- Agentic-tool registry refactor PR with green test suite.
- State and canonical migration helpers with explicit unit tests.

### 2) Antigravity SKILL.md backend (`antigravity_io.py`)

A new IO module mirroring `claude_io.py` / `codex_io.py` but only implementing the `skill` customization_type.

Antigravity's `SKILL.md` is the same open spec used by the project today (artifact metadata + Markdown body). The required fields are `name` and `description`. Optional fields per the open spec: `license`, `compatibility`, `metadata`, `allowed-tools`. Antigravity ignores any artifact metadata fields it does not recognize.

Module contents:

- `KNOWN_ANTIGRAVITY_FIELDS = {"customization_artifact_id", "name", "description", "license", "compatibility", "metadata", "allowed-tools"}`.
- `extract_customization_artifact_id_from_md` (reusable from `claude_io.py` via a shared artifact metadata-extraction helper).
- `parse_antigravity_skill_md(text, prior_canonical) -> canonical`: extracts known fields, packs unknown fields into `canonical["per_agentic_tool_extra"]["antigravity"]`. Body and `customization_artifact_id` round-trip via the shared artifact metadata mechanism.
- `render_antigravity_skill_md(canonical, prior_text=None) -> str`: emits `name`, `description`, `customization_artifact_id`, optional Antigravity-known fields from `canonical["per_agentic_tool_only"]["antigravity"]`, plus opaque passthrough from `canonical["per_agentic_tool_extra"]["antigravity"]`. Artifact metadata fields owned by Claude or Codex (`model`, `tools`, `hooks`, `mcpServers`) are **not** emitted on the Antigravity agentic_tool (Antigravity would ignore them but emitting them would muddy the user's artifact metadata).
- A note: Claude-specific fields preserved as opaque bytes in `per_agentic_tool_extra["claude"]` are never written to Antigravity. Round-trip stability requires that Antigravity parse + render is identity on the Antigravity-known subset.

Tests: parse/render round-trip; unknown-field passthrough; identity round-trip on real community SKILL.md samples (vendor a handful of public-domain fixtures into `tests/fixtures/antigravity/`).

Target modules: new `src/agents_sync/antigravity_io.py`; minor extraction of shared YAML artifact metadata helpers into a private module.

Deliverables:

- `antigravity_io.py` with parse + render + extract.
- Unit tests covering required-field-only, optional-fields, unknown-field-passthrough, BOM tolerance, CRLF tolerance, malformed-artifact-metadata handling.

### 3) Config and path defaults

Add Antigravity to the configuration model. Antigravity is registered like any other agentic_tool; if `antigravity_skills_dir` does not exist at startup or becomes missing later, Antigravity's status is `unavailable` per US-11 (registered, enabled, root unreachable). Claude<->Codex behavior is unaffected because removal-propagation never fires from an `unavailable` agentic_tool (US-11 AC-4).

Defaults:

| OS | Default `antigravity_skills_dir` |
|---|---|
| Linux | `~/.gemini/antigravity/skills/` |
| macOS | `~/.gemini/antigravity/skills/` |
| Windows | `%USERPROFILE%\.gemini\antigravity\skills\` |

Configuration:

- New config key `antigravity_skills_dir` (string path).
- New optional config key `antigravity_enabled` (bool, default `true`). Explicit `false` makes Antigravity's status `disabled` per US-11 (silent, never logged) — the only way to fully opt out. With `antigravity_enabled = true` and a missing directory, Antigravity's status is `unavailable` per US-11 (logged once on transition, retried each poll).
- New CLI flag `--antigravity-skills-dir`.

Validation:

- `validate_config` enforces structural validity (paths are well-formed, `name` is unique, the required IO functions exist, etc.) for every agentic_tool. Structural defects are configuration errors and cause the daemon to exit per US-07 AC-7.
- Whether a configured root exists, is readable, and is writable is a **runtime** question per US-11: a missing or unreadable root makes the agentic_tool `unavailable` but does not cause startup to fail. This rule is uniform across Claude, Codex, and Antigravity.

**Windows path-quirk handling:** there is a reported bug in Antigravity v1.19.6 on Windows where the global skills directory is read as `~/.gemini/antigravity/global_skills/` rather than `skills/`. The daemon does not attempt to auto-detect; users on affected versions override `antigravity_skills_dir` in config. Document this in the README under Notes.

Target modules: `config.py`, `cli.py`, `__main__.py`.

Deliverables:

- Updated `platform_defaults`, `merged_config`, `validate_config`.
- New CLI flag and arg-parser wiring.
- Tests for: enabled-but-missing-dir (status `unavailable`, logged once at startup, daemon continues), explicit-disable-with-existing-dir (status `disabled`, silent), default-when-dir-present (status `available`), explicit-override-path (honored).

### 4) Three-way sync algorithm activation

The N-way algorithm itself lands in workstream 1 (§1) and is exercised in Phase 1 against a participating-agentic_tool set of size 2. Workstream 4 covers the behaviors that only become observable once Antigravity is registered as a third participating agentic_tool and the set can reach size 3 — no new algorithm code, just new test cases.

The algorithm operates on the set of **participating** agentic_tools for a given customization_artifact (agentic_tools whose `supported_customization_types` include the customization_artifact's customization_type AND whose status is `available` per US-11).

Per-customization_artifact logic, after agentic_tool-registry refactor:

1. **Discovery** produces `CustomizationArtifactInfo.agentic_tools: dict[str, AgenticToolInfo]` mapping every agentic_tool that currently sees the customization_artifact_id to its `AgenticToolInfo`.
2. **Participating agentic_tools** = set of agentic_tools whose `supported_customization_types` include the customization_artifact's customization_type AND whose status is `available` per US-11. For agents, the candidate set is `{claude, codex}`; for skills, `{claude, codex, antigravity}` (when Antigravity is `available`).
3. **Present agentic_tools** = subset of participating agentic_tools where the file currently exists with the correct customization_artifact_id.
4. **Changed agentic_tools** = subset of present agentic_tools where the current digest differs from the agentic_tool's `last_written` in state.
5. Decision:
   - No present agentic_tools: orphan, drop from state.
   - Some present, some absent (relative to state): removal — see below.
   - All present, no changed agentic_tools: no-op.
   - All present, exactly one changed agentic_tool: project from that agentic_tool to all others.
   - All present, ≥2 changed agentic_tools: N-way conflict — pick `argmax(mtime)` over changed agentic_tools; archive the others' current bytes; project from winner to all others.
6. **Adoption** when state is empty for the customization_artifact:
   - One present agentic_tool: parse it, mint customization_artifact_id (if absent), inject customization_artifact_id on that agentic_tool, render to all other participating agentic_tools.
   - Multiple present agentic_tools without state: pick `argmax(mtime)` to be the authoritative parser; archive the others' current bytes; project from winner.

Removal semantics (the tricky case):

- "Removed on agentic_tool X" means: customization_artifact_id was in state for agentic_tool X, agentic_tool X was a participating agentic_tool this poll, and the artifact is no longer discovered (digest is None / file absent).
- Policy: if **any** participating agentic_tool has removed the artifact since last poll, treat removal as authoritative. For every surviving agentic_tool, **archive the current bytes via `archive.archive_move` before deleting**, then delete the artifact, then drop the customization_artifact from state. The archive step is non-skippable: if archiving any surviving agentic_tool fails (disk full, permission denied, etc.), abort the propagation for that customization_artifact — leave the surviving agentic_tools in place, log an error, retry on the next poll. State is only mutated after all surviving agentic_tools have been successfully archived AND removed.
- Order on each surviving agentic_tool: `archive_move(agentic_tool)` -> verify archive file exists and digest matches -> remove from agentic_tool -> only then proceed to the next agentic_tool. This makes partial-failure recoverable: if the daemon crashes mid-removal, the next poll re-runs the same logic against whatever remains.
- Also covers the agentic_tool that initiated the removal: if the user's `rm` raced with the poll and left a partial state (e.g., directory exists but `SKILL.md` is gone), the partial bytes on the initiating agentic_tool are **also** archived under the initiator's agentic_tool name before the state entry is dropped, so nothing the user briefly had on disk is lost.
- This preserves and strengthens the v0.2 semantic that removal is destructive **only after** archiving. It handles the N=3 case naturally: removing on one agentic_tool archives-then-removes the other two.
- Distinction vs. "agentic_tool was never present" (e.g. user just enabled Antigravity for the first time): that's adoption, not removal. The discriminator is `state[customization_artifact_id].agentic_tools[agentic_tool_name]` — present in state means previously seen, so absence in discovery is removal; absent from state means never seen, so the customization_artifact simply doesn't participate on that agentic_tool yet and the next render will fill it in.

Target-collision detection:

- Generalize `_block_target_collisions` to walk participating agentic_tools; today it checks `claude_path` and `codex_path` from state, tomorrow it walks `state[customization_artifact_id].agentic_tools`.
- Antigravity skill folder names (slugs) are subject to the same `target_slug` rules; reserved Windows basenames are already handled.

Target modules: `sync.py` (the rewrite lands in workstream 1 / Phase 1; workstream 4 is registry wiring plus the new 3-way test cases below).

Deliverables:

- `_process_pair` rewritten over participating agentic_tools.
- `_resolve_conflict_n_way` with `argmax(mtime)` and N-1 loser archives.
- `_propagate_removal` triggered by "previously in state, missing in discovery on any participating agentic_tool."
- New tests:
  - 3-way no-op poll.
  - Claude edit propagates to Codex and Antigravity.
  - Codex edit propagates to Claude and Antigravity.
  - Antigravity edit propagates to Claude and Codex.
  - 2 agentic_tools edit simultaneously (Claude + Codex newer than Antigravity): Antigravity gets the winner.
  - 3 agentic_tools edit simultaneously: latest mtime wins, other two are archived.
  - Removal on Antigravity wipes Claude and Codex **and an archive of the prior bytes exists for both surviving agentic_tools** under `archive/<customization_artifact_id>/claude/...` and `archive/<customization_artifact_id>/antigravity/...` (and `codex/...`); customization_artifact is dropped from state only after both archive files are present.
  - Symmetric: removal on Claude archives + removes Codex and Antigravity; removal on Codex archives + removes Claude and Antigravity.
  - Archive-failure short-circuit: if the surviving-agentic_tool archive write fails (simulated via monkeypatched `archive_move`), the surviving agentic_tools are **not** removed and the state entry is **not** dropped; daemon retries on the next poll.
  - First-boot adoption of an Antigravity-only new skill.
  - Adding Antigravity to an existing managed Claude+Codex customization_artifact (the artifact exists in state without an Antigravity entry; the next poll renders to Antigravity and updates state).

### 5) Adopting Antigravity into existing managed customization_artifacts

When Antigravity is newly added to the participating set for an already-managed Claude+Codex customization_artifact (e.g. user enables Antigravity for the first time, or re-enables it), the daemon must extend the customization_artifact to Antigravity without conflicts:

- If Antigravity has no artifact with that customization_artifact_id: render the canonical to Antigravity, update state.
- If Antigravity already has an artifact with that customization_artifact_id (e.g. user copied a SKILL.md manually): treat as a normal 3-way poll on the next cycle; if the Antigravity copy is newer than canonical, conflict resolution applies.
- If Antigravity has an unrelated artifact at the slug-derived target path: defer to §5.5's "Mixed group" rule — the managed customization_artifact wins, the unmanaged Antigravity file is archived under the managed `customization_artifact_id`, and the canonical is rendered over it. This is the project's loss-less posture applied uniformly; the prior bytes are recoverable from `archive/`. (Distinct from the multi-managed-id collision case at the same slug, which still falls through to `_block_target_collisions` per §5.5.)

Deliverable: explicit unit test for each of the three cases.

### 5.5) First-boot reconciliation of multi-tool new customization artifacts

#### Contract

Reconciliation operates **per customization_artifact, not per agentic_tool**. Each new customization artifact is matched to its peers by `(customization_type, target_slug(name))`, and each resulting group is decided in isolation. The end state on every participating agentic_tool is the **union** of distinct customization_artifacts observed across all agentic_tools, with two rules:

- **Duplicates (same artifact on ≥2 agentic_tools): newer wins.** `argmax(mtime)` decides; losers' bytes are archived; all agentic_tools converge to the winner's content under a single new `customization_artifact_id`.
- **Non-duplicates (artifact on exactly one agentic_tool): adopted as-is.** Each unique artifact keeps its own content, gets a fresh `customization_artifact_id`, and is propagated to every other participating agentic_tool.

There is no cross-artifact interaction. Two artifacts present on the same agentic_tool never influence each other's reconciliation. The total number of managed customization_artifacts after reconciliation equals the number of distinct `(customization_type, slug)` keys observed across all new customization artifacts that poll.

Concretely, if Claude has skills `{A, B}`, Codex has `{B, C}`, Antigravity has `{C, D}` — all new — the daemon ends up with four managed customization_artifacts (`A`, `B`, `C`, `D`), each present on all three agentic_tools. `B` and `C` are decided by mtime; `A` and `D` are simple adoptions.

#### Why this matters

The expected onboarding case for a v0.4 user: they have been maintaining their personal skill library independently on Claude, Codex, and (now) Antigravity. Most skills are duplicated across agentic_tools (often drifted in content). Some are unique to one tool. None of the copies carry a `customization_artifact_id`. The daemon must converge them into a coherent set of managed customization_artifacts on first boot — neither dropping unique skills nor producing N copies of the duplicates.

This is a distinct concern from §5 ("extending an already-managed 2-agentic_tool customization_artifact to Antigravity") because here **no artifact is managed yet** — every artifact on every participating agentic_tool is a new customization artifact. The v0.3 daemon handled this case poorly even in the 2-agentic_tool world: two same-slug new customization artifacts on Claude and Codex (neither carrying a `customization_artifact_id`) would each be adopted as its own managed artifact, then mutually collision-blocked. v0.4 fixes this for N=2 and N=3 with one algorithm.

#### Reconciliation algorithm

Run **before** per-agentic_tool adoption, on the set of new customization artifacts (`customization_artifact_id_present == False`) discovered this poll:

1. Compute a **reconciliation key** for every new customization artifact: `(customization_type, target_slug(name, customization_type))`. The slug already includes the customization_type suffix (`-agent` / `-skill`) and is normalized via the existing `slugify`, so it is comparable across agentic_tools regardless of original filename casing or extension.
2. Group new customization artifacts by reconciliation key.
3. For each group:
   - **Singleton group** (one agentic_tool has it): existing per-agentic_tool adoption path applies unchanged — mint customization_artifact_id, inject, project to other participating agentic_tools.
   - **Multi-agentic_tool group, all new**: **merge** path (see below).
   - **Mixed group: one or more agentic_tools also have a *managed* (customization_artifact_id-present) artifact with the same reconciliation key**: the managed customization_artifact wins. Archive each new artifact's bytes via `archive.archive_copy` under the managed customization_artifact's `customization_artifact_id` and that agentic_tool's name, then render the canonical to overwrite the new customization artifact. No new customization_artifact_id is minted. (This is just a beefier version of existing target-collision handling with archive-instead-of-block.)
   - **Multi-agentic_tool group with ≥2 *different* managed customization_artifact_ids at the same reconciliation key**: do **not** auto-merge across customization_artifact_ids — that would be lossy and irreversible. Fall through to existing `_block_target_collisions` behavior: block all involved customization_artifacts, log loudly, require user remediation (rename one of the conflicting artifacts).

#### Multi-agentic_tool merge path

When ≥2 participating agentic_tools hold new customization artifacts at the same reconciliation key:

1. Pick the **winner** = the agentic_tool with `argmax(mtime)` over the new customization artifacts in the group.
2. **Archive every agentic_tool's current bytes**, including the winner's. The winner's bytes will be re-written below with an injected `customization_artifact_id`, so the pre-injection bytes are archived under `archive/<new_customization_artifact_id>/<winner_agentic_tool>/<filename>.<ISO>` for symmetry with how the losers are archived. The losers are archived under their own agentic_tool names.
3. Mint a new `customization_artifact_id`.
4. Parse the winner's content into the canonical. **Do not** merge field-by-field from the losers — last-mtime-wins is the project's stated conflict policy and applying field-level merge here would diverge from `_resolve_conflict_n_way` semantics. The losers' content is recoverable from the archive; that is the project's chosen safety net.
5. Inject `customization_artifact_id` into the winner agentic_tool in place (existing adoption path).
6. Render canonical to every loser agentic_tool, overwriting the new customization artifact at the same path the loser already lived at (so user-visible filenames don't shuffle unless a slug normalization step demands it). Existing target-collision checks still run against unrelated unmanaged paths.
7. Write the new customization_artifact to state with all participating agentic_tools populated.

End state: one `customization_artifact_id`, one canonical, N agentic_tools in lock-step, N archive entries preserving everything that was on disk just before the merge.

#### Tie-breaking and edge cases

- **mtime tie** to second precision: deterministic tiebreaker is alphabetical by agentic_tool name (`antigravity` < `claude` < `codex`). Document this; it matters only for tests and post-mortems.
- **Body identical across agentic_tools, only `customization_artifact_id` missing**: detected as a no-content-change merge — archive all agentic_tools (cheap; identical bytes), pick any winner, inject customization_artifact_id. Still produces one customization_artifact.
- **New customization artifact on disabled agentic_tool**: not discovered, not considered. If the user later enables the agentic_tool, that agentic_tool then enters via the §5 "extend existing managed customization_artifact" path, not the merge path.
- **Merge with a slug-derived rename**: the winner's slug determines the canonical filename on each agentic_tool. If a loser's current filename differs from `target_slug(name)` after normalization (e.g. user had `Formatter.md` instead of `formatter.md`), the new file is written at the slug-normalized path and the original filename is the one archived. Avoid leaving two files on the same agentic_tool: the original is archived **then** removed before the new file is written. This is `stage_skill_dir`'s existing pattern, extended to the merge case.
- **Cross-customization_type same-slug**: skills and agents live in different directories and have different reconciliation keys (`customization_type` is part of the key). No collision risk.

#### Tests

Add explicit tests for each case:

- 2-agentic_tool new-artifact duplicate (Claude + Codex, no Antigravity participation, drifted content): produces one customization_artifact, both agentic_tools converge to winner content, loser agentic_tool has archive entry.
- 3-agentic_tool new-artifact duplicate: same, all three agentic_tools converge.
- **Mixed first-boot library:** Claude has `{A, B}`, Codex has `{B, C}`, Antigravity has `{C, D}`, all new. After one poll: exactly four managed customization_artifacts exist, all four skills present on all three agentic_tools, `B` resolved by mtime between Claude and Codex, `C` resolved by mtime between Codex and Antigravity, `A` and `D` adopted without conflict. No collision blocks. No skills lost.
- **Per-artifact independence:** in the mixed-library case, if `B` was newest on Claude and `C` was newest on Antigravity, the winner agentic_tool for `B` does not influence the winner agentic_tool for `C`. Verified by asserting both winners on different agentic_tools.
- Mixed managed + new at same slug: the agentic_tool carrying the new artifact adopts the managed canonical, the new bytes archived under the managed `customization_artifact_id`.
- Multi-customization_artifact_id collision at same slug: existing block-and-log behavior verified, no merge attempted.
- mtime-tie tiebreaker: deterministic outcome verified.
- Identical content across all agentic_tools for a duplicate new artifact: merge succeeds without content churn beyond customization_artifact_id injection; archives written (even though bytes are duplicates) so the contract is uniform.
- Merge winner's pre-injection bytes are archived alongside losers (symmetry property).
- Non-duplicate adoption preserves original filename when slug-normalized form already matches; otherwise renames and archives the original filename (parity with the existing adoption path).

#### Sequencing within the sync loop

Reconciliation runs as a new phase **between** discovery and `_block_target_collisions`:

```
discover() -> reconcile_new_groups() -> block_target_collisions() -> per-customization_artifact process()
```

The reconcile step rewrites the in-memory `customization_artifacts` mapping (collapsing multi-agentic_tool new-artifact groups into single merged entries) and then hands the resulting `customization_artifacts` to the existing collision check. After reconciliation, any remaining collisions are genuine (across different customization_artifact_ids or with truly unmanaged unrelated paths) and the existing block-and-log behavior is correct.

### 6) Archive and state directory layout

The current archive layout is:

```
archive/<customization_artifact_id>/<agentic_tool_name>/<filename>.<ISO-timestamp>
```

`<agentic_tool_name>` was always `claude` or `codex`. v0.4 adds `antigravity` as a valid `<agentic_tool_name>`. No layout change is required.

State `canonical/<customization_artifact_id>.json` schema:

- `schema_version: 2`.
- Replace `claude_only` / `codex_only` with `per_agentic_tool_only: {claude: {...}, codex: {...}, antigravity: {...}}`.
- Replace `claude_extra` / `codex_extra` analogously.
- Agentic-tool keys that have no content (empty dicts) may be omitted on serialization.

State `state.json` schema:

```json
{
  "schema_version": 2,
  "customization_artifacts": {
    "<customization_artifact_id>": {
      "customization_type": "skill",
      "agentic_tools": {
        "claude":      {"path": "...", "last_seen": "...", "last_written": "..."},

        "antigravity": {"path": "...", "last_seen": "...", "last_written": "..."}
      }
    }
  }
}
```

Reader accepts both v1 (unversioned, `claude_path` / `codex_path` flat keys) and v2; writer emits v2 only.

## Validation plan

### Functional checks

- Three-way propagation: edit on any agentic_tool, other two converge within two polls.
- Claude<->Codex agent flow unchanged.
- No-op polls remain no-op (zero writes).
- Conflict resolution: 2-way and 3-way both pick `argmax(mtime)`, losers archived.
- Removal: removing any one agentic_tool archives **every** surviving agentic_tool under `archive/<customization_artifact_id>/<agentic_tool_name>/` before deleting them; state is dropped only after all archive writes succeed; archive-write failures leave the surviving agentic_tools untouched and retry on the next poll.
- Adoption: a new Antigravity skill is adopted symmetrically.
- **First-boot reconciliation:** reconciliation is **per customization_artifact**. For each `(customization_type, slug)` group observed across new customization artifacts: duplicates (same artifact on ≥2 agentic_tools) are merged by `argmax(mtime)` with every agentic_tool archived; non-duplicates (artifact unique to one agentic_tool) are adopted as-is. End state on every agentic_tool is the **union** of distinct customization_artifacts observed — nothing dropped, no collision blocks, one managed customization_artifact per distinct `(customization_type, slug)`. Verified on a mixed first-boot library where Claude, Codex, and Antigravity each have a partly-overlapping set of new skills.
- Extension: existing 2-agentic_tool customization_artifact gains the third agentic_tool cleanly.

## Risks and mitigations

- **Risk:** the agentic_tool-registry refactor regresses two-agentic_tool behavior.
  - Mitigation: ship phase 1 with zero Antigravity code, keep the existing test suite as the contract.

- **Risk:** Antigravity ships breaking SKILL.md changes after v0.4 release (the open spec is community-maintained but not stable forever).
  - Mitigation: unknown artifact metadata fields ride through `per_agentic_tool_extra["antigravity"]` as opaque YAML. Schema is permissive on read.

- **Risk:** Antigravity later adds user-level agents with a stable spec, making the skills-only model awkward.
  - Mitigation: the `AgenticToolSpec.supported_customization_types` mechanism is already there; adding agents support becomes a new IO module + a one-element change to the agentic_tool's declared customization_types. No further sync-algorithm changes.

- **Risk:** the Windows `global_skills` quirk fragments the user base.
  - Mitigation: explicit configurable path; document the override; do not auto-detect (auto-detection would mask the underlying bug).

- **Risk:** user enables Antigravity for the first time with pre-existing skills on Claude+Codex, expecting one-shot propagation. Adoption logic must not see the new Antigravity directory as "everything was deleted on this agentic_tool."
  - Mitigation: the participating-agentic_tools + state-presence discriminator described in workstream 4 explicitly handles this case. Unit test it.

- **Risk:** user runs v0.4 for the first time with their personal skill library spread across all three agentic_tools — most skills duplicated (and drifted), some unique to one tool. Naive adoption would produce N customization_artifacts per duplicate and collision-block them; naive bulk reconciliation that just picks one source-of-truth agentic_tool would silently drop the unique skills.
  - Mitigation: §5.5 "First-boot reconciliation" treats each artifact independently by `(customization_type, slug)` group. Duplicates resolve by `argmax(mtime)` with every agentic_tool (winner included) archived before overwrite. Non-duplicates are adopted as-is and propagated to all agentic_tools. End state is the union of distinct customization_artifacts, one managed customization_artifact per `(customization_type, slug)`. Last-mtime-wins is consistent with the project's stated conflict policy; field-level merge is deliberately not attempted. Recovery path: losers' bytes are in `archive/<customization_artifact_id>/<agentic_tool_name>/` and can be reapplied manually if the user disagrees with a per-artifact mtime outcome.
