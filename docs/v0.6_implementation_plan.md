# v0.6 Implementation Plan — Canonical store as source of truth + cross-machine merge

**Implementation status (2026-06-02): built on `plan/v0.6-system-libraryfirst`.**
The codebase now has archive-before-drop canonicals, canonical heal/projection,
bulk-glitch recovery, FR-14 canonical digest detection, canonical-only import,
and cross-machine `last_modified_wins` merge. The remaining release work is
verification/CI and any follow-up audit fixes.

Implements the governance validated in
`docs/amendment/002-cross-machine-merge-import.md`: NFR-16/17, FR-12/13, US-12
AC-5/17/18/19, US-11 AC-8/AC-9, US-05 AC-5. The architecture §6.1 now documents
the implemented behaviour.

## 1. Principle and the one inversion

Today the daemon is **disk-driven**: `DiscoveryWalker` reads tool files, and a
managed pair that is absent from a tool is a *removal*. v0.6 makes the **canonical
store authoritative** (NFR-16): every tool file is a projection of the canonical,
and a tool that *should* have an artifact but lacks it is *healed* (re-projected),
unless the absence is a single deliberate deletion.

The whole feature reduces to **one new decision, evaluated per `(pair, tool)`**:

```
artifact recorded-or-derivable for (pair, tool) but absent on disk →
    never recorded for this tool          → HEAL  (import / new tool: project from canonical)
    recorded, and ≥2 of this tool's        → HEAL  (glitch: uninstall/unmount/mid-write; US-11 AC-9)
        artifacts vanished this poll
    recorded, and exactly 1 vanished        → DELETE (deliberate; propagate removal; US-11 AC-8)
```

Everything else (import becoming canonical-only, the cross-machine merge) composes
on top of this.

## 2. Target `sync_once` flow

```
validate_config; tool_status.refresh(); load_state
discovery, blocked = discovery.discover(state)
_reconcile_new_groups; block_target_collisions
── NEW P3: vanished = per available tool, {pair_id : state records tool but disk absent}
           glitch_tools = { tool : |vanished[tool]| >= 2 }
for pair_id, info in discovery:          # disk-present pairs
    process_pair(pair_id, info, state, glitch_tools)   # removal skips glitch tools
for pair_id in state \ discovery:        # disk-absent pairs (zero tools on disk)
    reconcile_absent_pair(pair_id, state, glitch_tools)   # P2/P3: heal vs delete vs preserve
── NEW P2: heal_pass — project every (managed pair, supporting available tool)
           that should hold the artifact but lacks it for a HEAL reason
save_state
```

`process_pair` keeps its current branches; only the removal branch consults
`glitch_tools`. The two new pieces are the **vanished pre-pass (P3)** and the
**heal pass (P2)**.

## 3. State schema

- An **import stub** is a managed pair with a canonical on disk and a `state`
  entry whose `agentic_tools` is **empty** (no tool recorded as holding it yet).
  `schema_version` does not change; an empty `agentic_tools` dict is already
  representable. The engine must stop assuming `len(agentic_tools) >= 1`.
- No new persisted field is required. "Never recorded for this tool" = `tool not
  in ps.agentic_tools`. "Vanished" = `tool in ps.agentic_tools` but absent in the
  poll's `info`. The glitch count is computed per poll, not stored.

## 4. Phased implementation

Each phase is independently testable and committable (full `pytest`, `mypy
--strict`, `ruff`). Order respects dependencies: P1 → P2 → P3 → P4 → P5.

### P1 — Archive a dropped canonical (US-05 AC-5)

- **`archive.py`**: add `archive_canonical(state_dir, pair_id)` → copies
  `canonical/<id>.json` to `archive/<id>/_canonical/<id>.json.<ISO-UTC>` (reuse the
  timestamp helper; `_canonical` is a reserved segment, never a tool name).
- **Call sites**: every place a canonical is dropped —
  `removal_propagator._drop_state_tools` (when `del state[pair_id]` empties the
  pair) and the import slug-displacement unlink (P5). Archive **before** unlinking
  the canonical file.
- **Tests**: dropping the last tool of a pair archives its canonical; archive path
  shape; failure-to-archive aborts the drop (NFR-01 / US-05 AC-4 parity).

### P2 — Project from canonical (heal); NFR-16 / US-11 AC-8 (never-recorded case)

- **`rendering.py`** already has `render_to_agentic_tool(..., prior_text=None)` —
  the projection primitive (import uses it today). No change beyond confirming it
  records the **post-write on-disk digest** in state (`update_state_n_way` already
  does `sha256_file(result.path)` — verify; this is the AC-8 / NFR-05 convergence
  guarantee).
- **`sync.py` heal pass** (new method `_heal_disk_absent`): for each managed pair
  in `state` whose canonical exists, for each enabled/supporting/**available** tool
  that (a) is not recorded in `ps.agentic_tools` (never projected) **or** (b) is a
  glitch-frozen absence (P3), and lacks the artifact on disk this poll → load
  canonical, `render_to_agentic_tool(prior_text=None)`, `update_state_n_way`. Skip
  tools whose root for the kind is unconfigured (reuse the kind-availability check).
- **Zero-disk pairs**: `reconcile_absent_pair` routes an import stub (empty
  `agentic_tools`) into the heal pass instead of `propagate_orphan_state`.
- **Tests**: a managed pair on tool A with a canonical, tool B available + never
  recorded → B receives the projection, digest recorded, **next `sync_once` is a
  no-op (NFR-05)**; an import stub (canonical + empty-tools state) projects to all
  supporting tools; an unconfigured-root tool is skipped, not crashed.

### P3 — Vanished pre-pass + heal-vs-delete + bulk-glitch guard (US-11 AC-8/AC-9, FR-04)

- **`sync.py`**: before the per-pair loop, compute
  `vanished: dict[tool, set[pair_id]]` from `state` × `discovery`
  (tool recorded for a pair but the pair/tool absent in the poll), restricted to
  `available` tools. `glitch_tools = {t for t,s in vanished.items() if len(s) >= 2}`.
- **Thread `glitch_tools` into `AdoptionEngine.process_pair`** (new parameter,
  default empty for callers/tests): in the removal branch, partition
  `missing_from_state` into `glitch` (tool ∈ `glitch_tools`) and `deliberate`
  (single). Propagate removal only for `deliberate`; leave `glitch` tools recorded
  in state (the P2 heal pass re-projects them). Emit one WARNING per glitch tool
  naming the count (US-11 AC-9).
- **`propagate_orphan_state`** (whole-pair absent): same partition — if the pair's
  tools are all glitch → preserve + heal; if a single deliberate removal → drop.
- **Tests**: 1 artifact removed from a tool → propagates (deletes elsewhere);
  2+ removed from one tool in one poll → none propagate, all re-projected next
  pass, library intact; removing the *last* single artifact still deletes it
  (closes the round-3 footgun); partial vs bulk on the same tool; glitch on tool A
  does not suppress a genuine single removal on tool B.

### P4 — Canonical-only import (US-12 AC-5, Decision A, FR-13)

- **`portable_archive.import_from_zip`**: delete the Phase-C
  `render_to_agentic_tool` loop. After staging+promoting canonicals (existing
  atomic Phase A/B), write a **per-artifact state stub** (`agentic_tools = {}`,
  carrying `last_modified`/`generation` from the zip). No tool roots touched.
  Per-artifact atomicity (FR-13): each `(canonical promote + stub)` lands or not;
  a failure leaves prior successes intact (the existing staging already gives
  canonical atomicity; extend the guarantee to the paired state stub).
- The next `sync_once` heal pass (P2) projects every stub → tools. **Daemon must be
  running** (or one `sync_once` invoked) for projection — update US-12 AC-5 prose
  and the README "Backup, Restore, Share" note accordingly.
- **Test migration** (breaking): existing US-12 AC-5/AC-11 tests assert import
  renders to tool roots before returning. Rewrite them to the canonical-only
  contract: after `import`, canonicals + stubs exist and no tool file is written;
  after one `sync_once`, every tool file appears and a second `sync_once` is a
  no-op. Audit `tests/test_cli_export_import.py`, `test_e2e_sync.py`, and any
  `portable_archive` tests.

### P5 — Cross-machine merge + preview + populated host (US-12 AC-17/18/19, FR-12)

- **`portable_archive._classify`**: replace the build-once slug index with an
  **incremental** one (seed from local state; update as each candidate is
  accepted). On a same-`(kind, slug)` collision (against local *or* an
  already-accepted import), decide the winner by **wall-clock `last_modified`**,
  ties → local, else → lexicographic `customization_artifact_id` (AC-17). Drop the
  `generation`-primary path for the import comparison (keep `generation` for the
  runtime daemon conflict path). Mark losers `retired`: not written to canonical or
  state, bytes archived under the winner's id (P1 `_canonical` archive).
- **`preview_import`**: report intra-import merges as well as local displacements
  (AC-18), before any disk write.
- **Populated-host (AC-19)** is then emergent: id-collision (AC-6) + slug-collision
  (AC-7/AC-17) reconcile to one pair per `(kind, name)`; displaced local canonicals
  archived (P1); next `sync_once` re-projects.
- **`state.py`**: drop the now-stale `generation`-as-import-tiebreaker docstring
  (the forward-pointer added in round 3 becomes the live description).
- **Tests**: the `tmp/dup_test.py` scenario (10 → 5, no crash) succeeds; pure
  cross-host tie resolves by lexicographic id; idempotent re-import (AC-11); import
  onto a populated host yields the union (AC-19); preview lists merges.

## 4b. Revision (2026-05-30) — canonical-aware daemon (FR-14)

Implementing P4 surfaced that canonical-only import collides with the daemon's
**tool-file digest** change detection: an import that replaces a *canonical*
(populated-host overwrite) is invisible to the daemon — the stale tool files match
state's `last_written`, so the import is never projected (or, if state is reset to
a stub, the stale files are re-adopted as the source, discarding the import). So
P4/P5 are re-scoped to make the daemon **canonical-aware** (FR-14) before import
becomes canonical-only.

**Revised P4 — canonical-aware daemon + migration:**
- **State schema 3 → 4:** add a per-pair `canonical_digest` (hash of the canonical
  at its last projection). `to_dict`/`from_dict` carry it; pre-1.0 cutover
  regenerates older files on first boot.
- **Migration:** on load of an older state, compute `canonical_digest` per managed
  pair from its on-disk canonical — establishing the canonical as the recorded
  source of truth.
- **`sync_once` canonical-first detection:** per pair, if the canonical's current
  digest != `ps.canonical_digest` and no tool changed → re-project the canonical to
  every supporting tool (FR-14), archiving displaced bytes (NFR-01), then refresh
  tool digests + `canonical_digest`. A tool-side change reverse-projects as today,
  then refreshes `canonical_digest`.
- **`project_from_canonical` archive-before-overwrite:** archive an existing
  tool-side file before re-projecting over it (NFR-01); stubs / glitch-vanished
  tools have no file, so they skip archiving.

**Revised P5 — canonical-only import + merge:** `import_from_zip` writes canonical
documents only, never state and never tool roots (AC-5 / FR-15). New imported
canonicals are adopted as orphan canonicals by the next `sync_once`; overwritten
canonicals re-project through the stale `canonical_digest` baseline. The slug
merge (AC-17), preview (AC-18), and populated-host union (AC-19) sit on top.

## 5. Cross-cutting verification

- **NFR-05 convergence** is the highest-risk invariant: every projection must
  record the post-normalisation on-disk digest so the next poll sees no delta. Add
  an explicit "import → `sync_once` → `sync_once` is a no-op" test and a
  "heal → `sync_once` no-op" test.
- **NFR-17 (unattended)**: a regression test driving add / edit / single-remove /
  bulk-remove / rename / tool-uninstall with **no CLI call** and asserting
  convergence.
- Full suite + `mypy --strict` + `ruff` green before each phase commit; the live
  daemon restarted on the branch and observed `failed=0` after P3 and P5.

## 6. Risks and order notes

- **P4 breaks existing import tests** — expected and scoped to P4; do not start P4
  until P2 (projection) is green, or the suite has no way to render imported stubs.
- **`tool_status` must not gain a `state` dependency** (a prior audit rejected
  that). The vanished/glitch computation lives in `sync.py` (after `load_state`),
  not in `ToolStatusTracker`.
- **Empty-`agentic_tools` pairs** must be tolerated everywhere that iterates state
  (discovery, removal, export). Grep for `next(iter(... agentic_tools))` and
  `assert ... agentic_tools` before P2.
- Keep each phase a single logical commit; restart the live daemon and re-run the
  arch audit after P3 (daemon inversion) and P5 (import) to catch coherence drift
  early.
