# agents_sync v0.4.1 - opencode Implementation Plan

This document defines the engineering plan for `v0.4.1`: a follow-up to the
v0.4 N-agentic-tool registry that adds **opencode** without changing the sync
algorithm or state schema.

The plan uses:

- `docs/v0.4_implementation_plan.md` as the primary format and architecture
  baseline.
- `docs/v0.2_implementation_plan.md` for the core sync invariants:
  `pair_id`, canonical JSON, daemon polling, archive-before-overwrite, and
  last-mtime-wins.
- `docs/v0.2.1_remediation_plan.md` for validation, duplicate-ID handling,
  and collision safety.
- `docs/v0.3_implementation_plan.md` for Windows paths, retries, LF stability,
  and reserved-name handling.
- `docs/opencode_integration_research.md` for opencode formats, paths,
  field mapping, and risks.

Terminology note: the opencode research note uses
`customization_artifact_id`; the current codebase uses `pair_id`. v0.4.1 keeps
`pair_id` and `state.json` schema version 2.

## Why v0.4.1

v0.4 generalized the sync engine to N registered agentic tools and added an
optional skills-only tool. v0.4.1 should use that extension point rather than
start a new architecture track.

One gap remains:

- opencode is absent, even though it exposes user-level agents and open-spec
  skills on disk.

v0.4.1 closes that gap:

- Agents gain opencode as a new participant.
- Skills gain opencode as a new participant.

## Scope

### In scope

- Add opencode as a registered agentic tool.
- opencode participates in:
  - `agent`: `~/.config/opencode/agents/*.md`
  - `skill`: `~/.config/opencode/skills/<name>/SKILL.md`
- Keep existing tool behavior unchanged.
- Preserve all v0.2/v0.2.1/v0.3/v0.4 safety invariants.
- Update config, CLI, installers, docs, and tests for opencode.

### Out of scope

- Project-scoped customizations.
- opencode commands, global rules, MCP servers, plugins, custom tools, themes,
  keybinds, credentials, session state, logs, or cache.
- Installing or updating opencode.
- State schema migration.
- Renaming `pair_id`.
- File watchers or field-level merge.

## Goals

1. Editing an opencode skill propagates to every available skill-capable peer
   within at most two polling intervals, and vice versa.
2. Editing an opencode agent propagates to every available agent-capable peer
   within at most two polling intervals, and vice versa.
3. Existing behavior is unchanged when opencode is disabled or unavailable.
4. No user-authored content is destroyed: every overwrite, merge, rename, or
   removal archives prior bytes first.
5. opencode-only metadata round-trips losslessly through
   `per_agentic_tool_only` / `per_agentic_tool_extra`.
6. Fields that a target cannot represent do not leak into that target's native
   output.

## Non-goals

- Do not rewrite `sync.py`'s N-way algorithm unless a small protocol gap is
  required for path-aware parsing.
- Do not translate opencode's full `permission` language into other tools.
  Preserve the raw object and only project simple safe allow/deny hints.
- Do not auto-detect every possible opencode Windows path. Pin the current
  default with `opencode debug paths` and keep explicit overrides.

## Baseline Findings

- v0.4 already has `AgenticToolSpec`, `CustomizationTypeIO`,
  `per_agentic_tool_only`, `per_agentic_tool_extra`, and N-way conflict/removal
  handling.
- opencode agent names come from the filename stem, but current parsers receive
  text only.
- opencode skills are open-spec `SKILL.md` folders; existing directory-skill
  staging can propagate auxiliary files.
- opencode Windows paths are uncertain in public docs and must be confirmed.
- Unknown top-level opencode frontmatter keys must be tested before `pair_id`
  injection is assumed safe.

## Target Matrix

opencode participates in both supported customization types:

| Customization type | opencode |
|---|:---:|
| `agent` | yes |
| `skill` | yes |

Default user-level roots:

## Workstreams

### 1) Pre-flight checks

- Test current opencode with an agent file containing top-level `pair_id` YAML
  frontmatter.
- Run `opencode debug paths` on Windows and pin the default root.

If opencode cannot safely ignore top-level `pair_id`, use one fallback:

- `options.agents_sync_pair_id`
- an HTML body comment
- a sidecar identity file

### 2) Minimal IO protocol extension

opencode agent identity comes from the filename stem. Add path-aware parsing
only if needed:

```python
parse(text, prior_canonical, *, artifact_path=None)
```

Existing parsers ignore `artifact_path`; opencode uses it for `name`.

Also decide whether opencode skills need a per-tool slug normalizer. If not,
fail closed before writing invalid opencode skill folders.

### 3) Add opencode IO

Target module:

- `src/agents_sync/opencode_io.py`

If the registry has moved to per-tool modules, use
`src/agents_sync/agentic_tools/opencode.py` instead.

#### Agents

Known opencode agent fields:

```python
KNOWN_OPENCODE_AGENT_FIELDS = frozenset({
    "pair_id",
    "description",
    "mode",
    "model",
    "temperature",
    "top_p",
    "steps",
    "maxSteps",
    "permission",
    "tools",
    "color",
    "hidden",
    "disable",
    "options",
})
```

Mapping:

- filename stem -> canonical `name`
- `pair_id`, `description`, body -> canonical identity/metadata/body
- `model` -> canonical `model` plus opencode provider hint when present
- `mode`, `temperature`, `top_p`, `steps`, `permission`, `color`, `hidden`,
  `disable`, `options` -> `per_agentic_tool_only["opencode"]`
- unknown frontmatter -> `per_agentic_tool_extra["opencode"]`
- deprecated `maxSteps` -> normalize to `steps`

Do not emit foreign fields into opencode output.

#### Skills

Known opencode skill fields:

```python
KNOWN_OPENCODE_SKILL_FIELDS = frozenset({
    "pair_id",
    "name",
    "description",
    "license",
    "compatibility",
    "metadata",
})
```

Mapping:

- `pair_id`, `name`, `description`, body -> canonical fields
- `license`, `compatibility`, `metadata` ->
  `per_agentic_tool_only["opencode"]`
- unknown frontmatter -> `per_agentic_tool_extra["opencode"]`
- auxiliary files -> existing `directory_skill` staging

Unit tests:

- agent and skill fixed-point round trips
- byte determinism
- `pair_id` extraction/injection/preservation
- unknown-field passthrough
- BOM and CRLF tolerance
- malformed YAML error
- model provider split/reattach
- raw `permission` preservation
- no foreign-field leakage

### 4) Config and registry

Add opencode defaults, CLI flags, and registry entry.

Config keys:

- `opencode_agents_dir`
- `opencode_skills_dir`
- `opencode_enabled` (default `true`)

CLI:

- `--opencode-agents-dir`
- `--opencode-skills-dir`
- `--opencode-enabled` / `--no-opencode-enabled`

Status behavior:

- Enabled + reachable roots -> `available`
- Enabled + missing/unreadable roots -> `unavailable`
- `opencode_enabled = false` -> `disabled`, silent, skipped

### 5) Matrix and safety tests

Agent matrix:

- opencode edit -> existing agent-capable peers converge
- existing agent-capable peer edit -> opencode converges
- new opencode agent adoption
- agent conflict with opencode involved: latest mtime wins, losers archived
- removal on any agent-capable tool archives/removes the available survivors

Skill matrix:

- opencode edit -> existing skill-capable peers converge
- existing skill-capable peer edit -> opencode converges
- skill conflict with opencode involved: latest mtime wins, losers archived
- first-boot mixed-library reconciliation includes opencode
- managed skill extends to opencode when opencode becomes available

Safety regressions:

- invalid `pair_id` skipped
- duplicate `pair_id` blocked
- target collision blocked before render
- archive failure aborts destructive removal
- no-op poll performs zero writes
- unavailable opencode never acts as a removal signal

## Phased Execution Plan

### Phase 1 - pre-flight and protocol fit

- Verify opencode `pair_id` frontmatter behavior.
- Verify opencode Windows paths.
- Add path-aware parse support / slug hook if required.

Exit criteria: current v0.4 suite remains green.

### Phase 2 - opencode IO

- Add opencode agent and skill parse/render/extract.
- Add isolated IO tests.

Exit criteria: opencode IO round trips pass before registry activation.

### Phase 3 - opencode config and registry

- Add opencode defaults, flags, and registry entry.
- Add available/unavailable/disabled tests.

Exit criteria: daemon starts cleanly with opencode disabled, unavailable,
empty, and populated.

### Phase 4 - integration matrix

- Add agent sync tests with opencode involved.
- Add skill sync tests with opencode involved.
- Add safety regression tests.

Exit criteria: full suite passes.

### Phase 5 - docs, installers, manual e2e

- Update docs and installer samples.
- Manual Linux e2e against a real opencode install.
- Manual Windows path check.

Exit criteria: docs reviewed, installer config validated, manual findings
recorded.

## Validation Plan

### Functional checks

- Agent sync:
  - opencode -> existing agent-capable peers
  - existing agent-capable peers -> opencode
- Skill sync:
  - opencode -> existing skill-capable peers
  - existing skill-capable peers -> opencode
- Adoption injects `pair_id`.
- Existing managed artifacts extend to newly available opencode roots.
- No-op polls perform no writes.

### Schema checks

- `state.json` remains schema version 2.
- Canonical documents continue to use `pair_id`.
- opencode data appears only under opencode passthrough bags.
- No v0.4.1 state migration is required.

### Cross-platform checks

- Linux/macOS:
  - `~/.config/opencode/agents`
  - `~/.config/opencode/skills`
- Windows:
  - opencode roots pinned by `opencode debug paths`
  - filesystem retry tests remain green

## Risks and Mitigations

- **Risk:** opencode rejects unknown frontmatter keys.
  - Mitigation: use the identity fallback selected in Phase 1.
- **Risk:** path-aware parsing causes churn.
  - Mitigation: keep it optional and regression-test existing tools.
- **Risk:** opencode skill slug rules are stricter than current slugification.
  - Mitigation: add per-tool slug normalization or fail closed before render.
- **Risk:** opencode Windows roots differ across versions.
  - Mitigation: document override paths and do not hide ambiguity.
- **Risk:** opencode native cross-read creates duplicate skills in its UI.
  - Mitigation: document opencode's cross-read disable flags; still write the
    native opencode root for symmetric sync.

## Done Criteria

`v0.4.1` is complete when all are true:

- Agents sync with opencode as an agent-capable participant.
- Skills sync with opencode as a skill-capable participant.
- opencode can be available, unavailable, disabled, and re-enabled safely.
- New opencode artifacts are adopted and receive `pair_id`.
- opencode-only fields round-trip losslessly.
- Cross-tool-only fields do not leak into incompatible outputs.
- Conflicts involving opencode, removal propagation, first-boot
  reconciliation, collision blocking, invalid-ID handling,
  archive-failure handling, and no-op behavior are covered by tests.
- All v0.4 tests and all new v0.4.1 tests pass.
- README, project docs, installer samples, and changelog are updated.
- Manual opencode e2e verifies that injected identity metadata does not break
  opencode loading.
