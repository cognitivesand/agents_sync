# v0.5 implementation plan - Cursor adapter

This plan implements the Cursor slice of `v0.5`, on the dedicated branch
`feat/v0.5-cursor`.

Cursor is the first real adapter that combine the existing customization_types with all three new v0.5 customization_types:

- `agent`, via `~/.cursor/agents/*.md` custom subagents
- `skill`, via `~/.cursor/skills/<name>/SKILL.md`
- `rules`, via `~/.cursor/rules/*.mdc`
- `slash_command`, via `~/.cursor/commands/*.md`
- `mcp_server`, via `~/.cursor/mcp.json[mcpServers]`

It does not touch Cursor's SQLite-backed settings state, cloud account state,
Notepads, Memories, Custom Modes, hooks, Background Agents, project-scoped
`.cursor/` files, or ignore files.

## Why Cursor

The v0.5 plan calls out Cursor as the first real consumer after the three new
customization_types land. That ordering is intentional: Cursor has large user
reach, but its syncable surface is only the local-file subset. Adding it proves
that the v0.4 registry model and the v0.5 type expansion can support a popular
IDE adapter without changing the sync engine.

Cursor's `agent` and `skill` support can be implemented with the existing v0.4
protocol surface, while `rules`, `slash_command`, and `mcp_server` require the
v0.5 type branches. The final Cursor PR should land after all five surfaces are
ready so the adapter ships as one coherent integration.

## Prerequisites

This PR starts only after these branches are merged or rebased in:

1. `feat/v0.5-rules`
2. `feat/v0.5-slash-command`
3. `feat/v0.5-mcp-server`

Required platform already provided by those branches:

- Existing `agent` single-file layout support.
- Existing `skill` directory layout support, including auxiliary files.
- `RulesFileLayout(extension=".mdc")`
- `SlashCommandFileLayout(extension=".md")`
- `SharedKeyedMapLayout(..., map_key_path=("mcpServers",), file_format="json")`
- Generic `rules_io`, `slash_command_io`, `mcp_server_io`, and open
  `SKILL.md` helpers
- `mcp_server_secret_policy`
- Slot-aware discovery, rendering, archiving, removal propagation, collision
  detection, and first-boot reconciliation

If Cursor implementation needs edits to `sync.py`, `state.py`, `adoption.py`,
`discovery.py`, `rendering.py`, or `archive.py`, pause and fix the protocol or
the type PR first. The Cursor PR should be adapter wiring plus Cursor-specific
IO, config, tests, and docs.

## Scope

### In scope

- Register a new `cursor` agentic_tool.
- Support Cursor custom subagents from:
  `~/.cursor/agents/*.md`.
- Support Cursor Agent Skills from:
  `~/.cursor/skills/<name>/SKILL.md`.
- Support `rules` from the user-level Cursor home:
  `~/.cursor/rules/*.mdc`.
- Support `slash_command` from:
  `~/.cursor/commands/*.md`.
- Support `mcp_server` slots from:
  `~/.cursor/mcp.json` under root key `mcpServers`.
- Per-OS defaults:

  | OS | Cursor home | Agents | Skills | Rules | Commands | MCP |
  |---|---|---|---|---|---|---|
  | Linux | `~/.cursor/` | `~/.cursor/agents/` | `~/.cursor/skills/` | `~/.cursor/rules/` | `~/.cursor/commands/` | `~/.cursor/mcp.json` |
  | macOS | `~/.cursor/` | `~/.cursor/agents/` | `~/.cursor/skills/` | `~/.cursor/rules/` | `~/.cursor/commands/` | `~/.cursor/mcp.json` |
  | Windows | `%USERPROFILE%\.cursor\` | `%USERPROFILE%\.cursor\agents\` | `%USERPROFILE%\.cursor\skills\` | `%USERPROFILE%\.cursor\rules\` | `%USERPROFILE%\.cursor\commands\` | `%USERPROFILE%\.cursor\mcp.json` |

- `cursor_enabled` config key, default `true`.
- Runtime `unavailable` status when a configured Cursor root is missing.
- Secret handling for MCP servers through the global
  `mcp_server_secret_policy`.
- Unit and integration tests covering all five supported types.
- README, protocol notes, installer sample config, and requirements matrix
  updates.

### Out of scope

- Project-scoped files: `<project>/.cursor/rules/`,
  `<project>/.cursor/commands/`, `<project>/.cursor/mcp.json`,
  `<project>/.cursor/agents/`, `<project>/.cursor/skills/`,
  `<project>/.agents/skills/`, `<project>/.cursorrules`, project `AGENTS.md`,
  Cursor Background Agent files, and project ignore files.
- Cursor SQLite / VS Code global state: `state.vscdb`, in-app User Rules,
  Team Rules, model choices, privacy toggles, account settings, Notepads, and
  UI-created Custom Modes.
- Cursor hooks. They need the future `hooks` customization_type because hook
  scripts must be modeled as auxiliary files.
- Cursor `modes.json`. It belongs to the deferred `mode` customization_type.
- Key-level sync of Cursor settings in `settings.json`.

## Goals

1. A Cursor custom subagent syncs to every other available tool that supports
   `agent`, and agents from other tools render into Cursor's subagent format.
2. A Cursor Agent Skill syncs to every other available tool that supports
   `skill`, including auxiliary files next to `SKILL.md`.
3. A rule edited in Cursor user files syncs to every other available tool that
   supports `rules`, and rules from other tools render as `.mdc` files for
   Cursor.
4. A Cursor slash command syncs to every other available tool that supports
   `slash_command`, preserving the body byte-for-byte.
5. A Cursor MCP server slot syncs through `mcp_server` without touching sibling
   slots in `~/.cursor/mcp.json`.
6. Cursor can be enabled on a machine without Cursor installed; missing roots
   make the tool `unavailable`, not fatal.
7. Cursor's unsupported and cloud/SQLite-only surfaces are documented clearly
   so users do not expect `agents_sync` to move them.

## Non-goals

- Auto-detect Cursor installation or profile-specific VS Code paths.
- Sync project-scoped Cursor customizations.
- Translate Cursor Custom Modes into agents. Cursor subagents are supported;
  UI custom modes remain a different, deferred customization_type.
- Persist identity outside the managed artifacts.
- Add new sync-engine behavior.

## Baseline findings

- Cursor 2.4 introduced custom subagents and Agent Skills in the editor and
  CLI, with `SKILL.md`-based skills and file-backed subagent definitions.
- User-level Cursor subagents are expected under `~/.cursor/agents/*.md`.
  Project-level `.cursor/agents/` is intentionally out of scope.
- User-level Cursor skills are expected under
  `~/.cursor/skills/<name>/SKILL.md`. Project-level `.cursor/skills/` and
  `.agents/skills/` are intentionally out of scope.
- Cursor rules are `.mdc` Markdown files with YAML frontmatter. `description`,
  `globs`, and `alwaysApply` are meaningful Cursor fields.
- Cursor rule type is derived from `alwaysApply`, `globs`, and `description`;
  it is not a separate field.
- Cursor slash commands are plain Markdown files. Cursor does not document
  frontmatter for them.
- Cursor MCP servers live in a JSON shared keyed map at
  `~/.cursor/mcp.json[mcpServers]`.
- Cursor CLI and Cloud Agents have had parity gaps around user-level subagents
  and skills. The adapter still writes the documented user-level file surfaces;
  release validation must include a manual Cursor IDE and CLI smoke test.
- Cursor's highest-risk surfaces are intentionally out of sync scope because
  they live in SQLite or cloud state.

## Resolved design decisions for this PR

- **D-cur-1. User scope only.** The adapter reads and writes only the
  user-level `~/.cursor/` home. Project-scope files remain owned by the
  user's repositories.
- **D-cur-2. Supported types.** Cursor declares exactly
  `{agent, skill, rules, slash_command, mcp_server}`.
- **D-cur-3. Agent identity.** Cursor subagents are Markdown files under
  `~/.cursor/agents/*.md`. The adapter uses YAML frontmatter with `pair_id`,
  keeps common canonical fields (`name`, `description`, `model`, `tools`,
  `body`) portable, and stores Cursor-only prompt/tool/model knobs under
  `per_agentic_tool_only["cursor"]`.
- **D-cur-4. Skill identity.** Cursor skills use the open `SKILL.md` directory
  shape under `~/.cursor/skills/<name>/SKILL.md`. The adapter shares the same
  open-spec helper used by Claude/Codex/Antigravity/OpenCode wherever possible,
  with Cursor-specific optional fields preserved in
  `per_agentic_tool_only["cursor"]` or `per_agentic_tool_extra["cursor"]`.
- **D-cur-5. Rule identity.** Cursor rules store `pair_id` in `.mdc`
  frontmatter. Cursor should ignore unknown frontmatter fields; tests verify
  round-trip stability.
- **D-cur-6. Slash-command identity.** Cursor slash commands stay plain
  Markdown. The adapter stores `pair_id` in a first-line HTML comment:
  `<!-- agents_sync:pair_id=<uuid> -->`. The parser strips this comment from
  canonical `body`; the renderer re-injects it. This avoids YAML frontmatter
  that Cursor would likely treat as command text.
- **D-cur-7. MCP identity.** Cursor MCP slots use the shared v0.5
  `mcp_server_io` convention: top-level `pair_id` inside the slot, with the
  slot rendered through the JSON shared-keyed-map path.
- **D-cur-8. Recursive command names.** `~/.cursor/commands/git/commit.md`
  maps to canonical command name `git:commit`, matching the generic
  `slash_command` namespacing rule. If Cursor only documents flat commands,
  recursive support remains harmless and improves cross-tool parity.
- **D-cur-9. Skill nesting.** Cursor skills render one folder deep:
  `~/.cursor/skills/<slug>/SKILL.md`. Nested category folders are not used
  because Cursor IDE/CLI/Automations have shown inconsistent behavior for
  nested skills.
- **D-cur-10. Secret policy.** Cursor MCP secrets are handled only by
  `mcp_server_secret_policy`; the Cursor adapter does not invent a separate
  policy.

## Workstreams

### 1) Cursor IO module

Target module: new `src/agents_sync/cursor_io.py`.

Implement these helpers:

- `extract_pair_id_from_cursor_agent_md(text) -> str | None`
- `parse_cursor_agent_md(text, prior_canonical, *, artifact_path, artifact_root)`
- `render_cursor_agent_md(canonical, prior_text=None) -> str`
- `extract_pair_id_from_cursor_skill_md(text) -> str | None`
- `parse_cursor_skill_md(text, prior_canonical, *, artifact_path, artifact_root)`
- `render_cursor_skill_md(canonical, prior_text=None) -> str`
- `extract_pair_id_from_cursor_rule_mdc(text) -> str | None`
- `parse_cursor_rule_mdc(text, prior_canonical, *, artifact_path, artifact_root)`
- `render_cursor_rule_mdc(canonical, prior_text=None) -> str`
- `extract_pair_id_from_cursor_command_md(text) -> str | None`
- `parse_cursor_command_md(text, prior_canonical, *, artifact_path, artifact_root)`
- `render_cursor_command_md(canonical, prior_text=None) -> str`
- Cursor MCP wrappers around shared `mcp_server_io` with Cursor's JSON dialect.

Agent behavior:

- Parse Cursor subagents as Markdown with YAML frontmatter plus body.
- Derive `name` from the filename stem when available.
- Round-trip common fields: `name`, `description`, `model`, `tools`, and body.
- Preserve Cursor-only fields such as subagent prompt/tool/model settings under
  `per_agentic_tool_only["cursor"]` until the exact public schema is pinned by
  fixtures.
- Preserve unknown frontmatter in `per_agentic_tool_extra["cursor"]`.
- Render to `~/.cursor/agents/<target_slug>.md`.

Skill behavior:

- Reuse the shared open-spec `SKILL.md` parser/renderer where possible.
- Render to `~/.cursor/skills/<target_slug>/SKILL.md`.
- Preserve auxiliary files exactly like existing skill adapters.
- Round-trip `name`, `description`, `license`, `compatibility`, `metadata`,
  and Cursor-specific optional fields.
- Do not render nested skill category directories.

Rules behavior:

- Reuse `rules_io` where possible.
- Use `.mdc` suffix.
- Derive `name` from the file stem or relative path.
- Preserve body verbatim.
- Round-trip `description`, `globs`, and `alwaysApply`.
- Store Cursor-only derived rule type diagnostics, if useful, under
  `per_agentic_tool_only["cursor"]`, but do not render a synthetic type field.
- Preserve unknown frontmatter in `per_agentic_tool_extra["cursor"]`.

Slash-command behavior:

- Treat the whole file as body except for the leading identity comment.
- Derive `name` from the relative file path; subdirectories become `:`
  namespaces.
- Do not render YAML frontmatter.
- Do not infer `description` from headings in the first implementation.
  Keeping the body byte-stable matters more than being clever.
- Preserve interpolation text exactly as written.

MCP behavior:

- Parse Cursor's `type` field:
  - missing `type` with `url` means canonical `transport = "streamable-http"`;
  - `type = "stdio"` means stdio;
  - `type = "sse"` means SSE;
  - known HTTP spellings normalize to canonical transport names.
- Preserve unknown server fields in `per_agentic_tool_extra["cursor"]`.
- Let shared secret-policy code handle `env`, `headers`, and `auth`.

Deliverables:

- New Cursor IO module.
- Focused IO unit tests for agents, skills, rules, commands, and MCP slot
  dialect.

### 2) Agentic-tool spec registration

Target module: `src/agents_sync/agentic_tool_spec.py`.

Add `_build_cursor_spec()`:

```python
return AgenticToolSpec(
    name="cursor",
    config_dir_keys={
        "agent": "cursor_agents_dir",
        "skill": "cursor_skills_dir",
        "rules": "cursor_rules_dir",
        "slash_command": "cursor_commands_dir",
        "mcp_server": "cursor_mcp_servers_file",
    },
    io={
        "agent": CustomizationTypeIO(
            parse=parse_cursor_agent_md,
            render=render_cursor_agent_md,
            extract_pair_id=extract_pair_id_from_cursor_agent_md,
            storage="single_file",
            file_suffix=".md",
        ),
        "skill": CustomizationTypeIO(
            parse=parse_cursor_skill_md,
            render=render_cursor_skill_md,
            extract_pair_id=extract_pair_id_from_cursor_skill_md,
            storage="directory_skill",
            file_suffix="",
        ),
        "rules": CustomizationTypeIO(
            parse=parse_cursor_rule_mdc,
            render=render_cursor_rule_mdc,
            extract_pair_id=extract_pair_id_from_cursor_rule_mdc,
            file_layout=RulesFileLayout(extension=".mdc"),
            recursive=True,
        ),
        "slash_command": CustomizationTypeIO(
            parse=parse_cursor_command_md,
            render=render_cursor_command_md,
            extract_pair_id=extract_pair_id_from_cursor_command_md,
            file_layout=SlashCommandFileLayout(extension=".md"),
            slugify_name=slash_command_slug,
            recursive=True,
        ),
        "mcp_server": CustomizationTypeIO(
            parse=parse_cursor_mcp_server_json,
            render=render_cursor_mcp_server_json,
            extract_pair_id=extract_pair_id_from_cursor_mcp_server_json,
            file_layout=SharedKeyedMapLayout(
                shared_path_config_key="cursor_mcp_servers_file",
                map_key_path=("mcpServers",),
                file_format="json",
            ),
        ),
    },
    disable_config_key="cursor_enabled",
)
```

Then add `"cursor": _build_cursor_spec()` to `default_agentic_tools()` in
alphabetical order.

Deliverables:

- Cursor present in the default registry.
- `tests/test_agentic_tool_spec.py` asserts its supported types, file suffixes,
  recursive flags, and config keys.

### 4) Runtime availability behavior

Target modules: no sync-engine changes expected.

Expected runtime semantics:

- If `cursor_enabled = false`, Cursor is `disabled` and silent.
- If all configured Cursor paths are missing, Cursor is `unavailable`.
- If some roots exist, Cursor participates only for the available
  customization_types.
- A missing `cursor_mcp_servers_file` means no Cursor MCP slots discovered; it
  must not make agents, skills, rules, or slash commands unavailable.
- A newly created MCP server from another tool creates `~/.cursor/mcp.json`
  with a root `mcpServers` map.

## Risks and mitigations

- **Risk:** Cursor commands treat metadata as prompt text.
  - Mitigation: use a compact HTML identity comment instead of YAML
    frontmatter, strip it from canonical body, and document the convention.

- **Risk:** Cursor subagent schema changes or differs between IDE and CLI.
  - Mitigation: keep unknown frontmatter in `per_agentic_tool_extra["cursor"]`,
    pin tests to real Cursor fixtures, and include manual IDE/CLI validation.

- **Risk:** Cursor skills are discovered inconsistently across IDE, CLI,
  Automations, SSH, or nested folders.
  - Mitigation: render only the documented one-folder user-level layout
    `~/.cursor/skills/<slug>/SKILL.md`; avoid nested category directories and
    symlink assumptions.

- **Risk:** Cursor rejects unknown MCP slot keys such as `pair_id`.
  - Mitigation: keep this aligned with the v0.5 MCP identity convention and
    add a manual Cursor smoke test before release. If Cursor rejects unknown
    keys, move identity to a supported comment/metadata carrier in the MCP type
    PR rather than hacking only Cursor.

- **Risk:** users expect in-app User Rules to sync.
  - Mitigation: README and project description explicitly say the in-app pane
    is SQLite/cloud-backed and out of scope; users should use
    `~/.cursor/rules/*.mdc`.

  - Mitigation: keep v0.5 consistent with the project-wide user-level scope.

- **Risk:** MCP files contain literal secrets.
  - Mitigation: rely on `mcp_server_secret_policy`, default `refuse`.
